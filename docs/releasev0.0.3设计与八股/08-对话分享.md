# 对话分享 — 设计与八股（后端）

> V0.0.3 的传播型特性：把一段对话生成一个**快照式只读分享链接**，任何人（无需登录）打开链接都能看到当时对话的内容；原对话后续继续聊、改名、删除都不影响已分享的快照。支持自定义标题、设置过期（永久 / 7 天 / 30 天）、取消分享。头像与图片在创建时就转成 base64 data URL 冻结进快照，公开页无需鉴权即可直接渲染。本篇只讲后端。

---

## 一、功能定位与需求

- **快照式分享**：创建分享时，把当前会话的消息脱敏、冻结成一份快照存库。之后原会话怎么变都和分享内容无关——分享的是「那一刻的对话」。
- **公开只读访问**：凭随机不可猜的 `share_token` 访问公开页，**不需要登录**。只读，不能在分享页继续聊。
- **同会话复用**：同一个会话已有有效分享时，再次分享只刷新该链接的快照（标题/过期/头像一起刷新），不重复生成新链接，避免一个会话散落一堆链接。
- **可管理**：「我的分享」列表、改标题、取消（软删，保留痕迹）、设过期。
- **脱敏**：快照只保留 user/assistant 的文本与图片，剔除工具调用、引用、记忆等内部细节。
- **群聊兼容**：群聊会话也能分享，快照按发言人扩展（每条带 `sender_name` / `sender_avatar`），公开页能像微信群一样区分不同角色。

## 二、数据模型与迁移

### 2.1 表 `conversation_shares`（`api/app/models/conversation_share_model.py`）

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | UUID 主键 | |
| `user_id` | UUID，外键 users，级联删，带索引 | 分享归属用户，数据隔离 |
| `conversation_id` | UUID，带索引 | 来源会话；**仅记录、不设强外键约束**——会话被删不应连带删快照 |
| `share_token` | String(64)，唯一，带索引 | 公开访问令牌，随机不可猜 |
| `title` | String(256) | 分享标题（默认取会话标题） |
| `snapshot` | JSONB，默认 `[]` | 消息快照 `[{role, content, images?, sender_name?, sender_avatar?}]` |
| `user_avatar` | Text，可空 | 用户头像 data URL（公开页直接用） |
| `ai_avatar` | Text，可空 | 当前生效 AI 角色头像 data URL |
| `ai_name` | String(64)，可空 | AI 角色名 |
| `is_active` | Boolean，默认 true，带索引 | 是否有效；取消即置 false（保留痕迹，不物理删） |
| `expire_at` | DateTime(tz)，可空 | 过期时间；空=永久 |
| `view_count` | Integer，默认 0 | 浏览次数 |
| `created_at` / `updated_at` | DateTime(tz) | |

设计要点：
- **`conversation_id` 不设外键级联**：快照是独立冻结的数据，来源会话删除后分享仍应可访问，所以只存 id 作为弱引用，不让数据库强绑。
- **头像/图片不存 file_key 而存 data URL**：公开页无登录态、拿不到带鉴权的文件接口，所以创建时就把头像、图片读出来压缩成 base64 data URL 内联进快照，公开页直接 `<img src="data:...">` 渲染，零鉴权依赖。

### 2.2 迁移链（`d429ecd1d3a9` → `9596cfc27bc1`）

- **`d429ecd1d3a9`**（create conversation_shares）：建表 + 四个索引（`user_id` / `conversation_id` / `is_active` / 唯一的 `share_token`）。
- **`9596cfc27bc1`**（share add avatars）：补加 `user_avatar` / `ai_avatar` / `ai_name` 三列（都可空，纯 add column，无回填问题）。

> **迁移链教训**：这条特性开发期间迁移链一度出现过空迁移（autogenerate 生成了没有实际变更的空 revision）导致 `down_revision` 指向混乱，最终靠手动核对 `revision` / `down_revision` 串联关系修正。经验是：**每次 autogenerate 后必须打开脚本核对 upgrade/downgrade 是否为空、down_revision 是否接在正确的前驱上**，不要盲目 `upgrade head`。

## 三、核心实现与代码路径

分层：`conversation_share_controller.py`（路由，含公开无鉴权路由）→ `conversation_share_service.py`（快照冻结 + 头像/图片 data URL + 复用/取消/公开查看）→ `conversation_share_repository.py`（数据访问，带 user_id 隔离 + 按 token 查）。

### 3.1 公开路由无鉴权（`conversation_share_controller.py`）

控制器拆成两个 router：

```python
router = APIRouter(tags=["share"])                       # 鉴权路由
public_router = APIRouter(prefix="/public", tags=["share-public"])  # 公开路由
```

- **鉴权路由**（都带 `Depends(get_current_user)`）：
  - `POST /conversations/{conversation_id}/share`：创建/刷新分享。
  - `GET /shares`：我的分享列表。
  - `DELETE /shares/{share_id}`：取消分享。
- **公开路由**（`public_router`，**没有 `get_current_user`**）：
  - `GET /public/shares/{token}`：凭 token 查看分享内容，任何人可访问。

公开路由刻意不挂鉴权依赖，这是「公开页无需登录就能看」的关键。安全性靠 `share_token` 的不可猜性（`secrets.token_urlsafe(16)`）+ `is_active` + `expire_at` 三道校验保证，而不是靠登录态。

### 3.2 快照冻结（`_build_snapshot`）

`create_share` 时调用 `_build_snapshot`，把会话消息脱敏冻结：

- 只保留 `role in (user, assistant)` 的消息，剔除 system 等。
- 每条只取 `content` 文本 + 图片（`images`），**不带**工具调用、引用、记忆等内部 meta。
- 既无文字又无图的消息跳过；纯图片（无文字）消息保留。
- **群聊扩展**：assistant 消息若有 `sender_persona_id`，额外带 `sender_name`（从 `meta_data.sender_name` 读）和 `sender_avatar`（按角色卡取头像 data URL，用 `avatar_cache` 字典按 persona_id 缓存，避免同一角色反复压缩）。

### 3.3 头像与图片转 base64 data URL（公开页免鉴权显示）

公开页没有登录态，无法访问受鉴权保护的 `/files/{key}`。所以分享创建时就把头像、图片读出来、压缩、转 data URL 冻结进快照，公开页直接内联渲染。四个工具方法（均在 service 内）：

- **`_avatar_data_url(file_key)`**：头像 file_key → 读存储 → `_compress_avatar` 压缩 → base64 → `data:image/jpeg;base64,...`。读取/压缩失败返回 `None`，前端回退默认头像（不阻断）。
- **`_compress_avatar(raw)`**（静态方法）：PIL 处理——透明通道贴白底转 RGB → **中心裁剪成正方形** → 缩到 **96px** → JPEG（quality=82）。头像展示才 34px，96px 足够清晰，体积极小，无需大小门控。失败回退原图。
- **`_image_data_url(file_key)`**：消息图片 → 读存储 → 透明贴白底 → **等比缩放（长边 ≤1024）** → JPEG（quality=80）→ data URL。
- **`_persona_avatar_data_url(persona_id)`**：群聊用，按角色卡 id 取 `avatar_key` 再走 `_avatar_data_url`。

`_snapshot_images` 单条消息最多取 9 张图，逐张 try/except，失败的跳过不影响其余（局部降级）。

创建分享时还会解析头像三件套：用户头像（`User.avatar`）、当前生效角色（`persona_repo.get_active`）的 `name` 与 `avatar_key`，全部转 data URL 存进 `user_avatar` / `ai_avatar` / `ai_name`。整段解析包在 try/except 里，失败只 warning 不阻断分享创建。

### 3.4 同会话复用刷新（`create_share`）

```python
existing = await self.repo.get_active_by_conversation(user_id, conversation_id)
if existing:
    # 复用：刷新快照、标题、过期、头像，不新建链接
    existing.snapshot = snapshot
    existing.title = share_title
    existing.expire_at = expire_at
    existing.user_avatar = user_avatar
    ...
    return await self.repo.save(existing)
# 否则新建
share = ConversationShare(..., share_token=secrets.token_urlsafe(16), ...)
```

`get_active_by_conversation` 查该会话的有效分享（`is_active=true`）。有就刷新它的快照（让链接内容跟上最新对话），URL 不变；没有才生成新 token 建链接。这样一个会话最多一条有效分享，避免链接泛滥。

创建前还校验：会话存在（否则 `BizError("会话不存在", 4070, 404)`）、快照非空（空对话不能分享，`4071`）。

### 3.5 取消（软删，保留痕迹）

`revoke` 不物理删除，只置 `is_active = false`：

```python
share.is_active = False
await self.repo.save(share)
```

保留行便于审计与「我的分享」里展示历史状态；公开访问时 `is_active=false` 会被 `get_public` 拦下。

### 3.6 公开查看（`get_public`）+ 过期判断 + 浏览数

`GET /public/shares/{token}` → `get_public`：

1. 按 token 查 → 不存在或 `is_active=false` → `BizError("分享不存在或已取消", 4073, 404)`。
2. **过期判断**：`expire_at` 非空且早于当前时间 → `BizError("分享链接已过期", 4074, 404)`。注意处理时区：库里取出的 `expire_at` 若 `tzinfo is None` 先补 `timezone.utc` 再比较，避免「naive vs aware datetime」比较报错。
3. **浏览数 +1**：`view_count += 1` 保存，失败只 warning 不影响查看（副作用不阻断主流程）。
4. 返回脱敏快照 + 头像三件套 + 标题 + 创建时间。

### 3.7 过期策略

`ShareCreateRequest.expire_days`：`None`=永久；传 7 / 30 等正整数则 `expire_at = now(utc) + timedelta(days=expire_days)`。过期不物理删，仅在公开查看时拦截（返回 404 语义的 `4074`）。

## 四、设计取舍（已定决策）

| 决策 | 选择 | 理由 |
|------|------|------|
| 分享形态 | **快照式冻结**，非实时镜像 | 分享「那一刻的对话」，原对话后续变化/删除不影响；也避免实时拉取时的鉴权与一致性问题 |
| 公开访问鉴权 | **public_router 无 `get_current_user`** | 「无需登录就能看」是核心需求；安全靠不可猜 token + is_active + 过期三道校验 |
| 头像/图片 | 创建时转 **base64 data URL** 冻结 | 公开页无登录态拿不到鉴权文件接口，内联 data URL 才能免鉴权直接渲染 |
| 头像压缩 | 96px 中心裁剪 JPEG | 展示仅 34px，96px 足够，体积极小，避免快照过大 |
| 图片压缩 | 长边 1024 JPEG | 控制快照 JSONB 体积，清晰度够用 |
| 同会话重复分享 | **复用并刷新**，不新建 | 一个会话最多一条有效分享，避免链接泛滥 |
| 取消方式 | **软删 `is_active=false`** | 保留痕迹便于审计与列表展示，不物理删 |
| `conversation_id` | 弱引用、**不设外键级联** | 会话删除后分享快照仍应可访问 |
| 过期 | 永久 / 7 天 / 30 天 | 简单够用；过期不删，查看时拦截 |

## 五、易踩坑点

1. **公开页拿不到鉴权文件接口**：这是整个特性最核心的坑。如果快照里只存 file_key，公开页（无登录态）请求 `/files/{key}` 会被 401 拦掉，头像图片全裂。解决办法是创建分享时就把头像/图片读出来压缩成 data URL 内联进快照，公开页零鉴权直接渲染。代价是快照体积变大，所以必须配合压缩（头像 96px、图片长边 1024、JPEG）。

2. **过期时间的时区比较**：库里的 `expire_at` 可能是 naive datetime，直接和 `datetime.now(timezone.utc)`（aware）比较会抛 `TypeError`。`get_public` 里先判断 `exp.tzinfo is None` 就补 `timezone.utc` 再比，避免崩溃。

3. **`conversation_id` 不能设级联外键**：快照要独立于来源会话存活。若设了 `ondelete=CASCADE`，删会话会连带删掉已分享的快照，分享链接突然失效，违背「分享的是那一刻对话」的语义。所以只存 id 弱引用。

4. **迁移空 revision 混乱**：autogenerate 偶尔生成 upgrade/downgrade 为空的迁移，或 down_revision 串错，导致 `upgrade head` 行为异常。每次生成后必须人工核对脚本内容与 down_revision 链路，发现空迁移/错链手动修正。

5. **副作用失败不能阻断主流程**：浏览数 +1、头像解析、单张图片转换都包了 try/except 只记 warning。一张图压缩失败不应让整个分享创建或公开查看失败——逐项降级。

6. **群聊快照的发言人头像缓存**：群聊一条消息一个发言人，若不缓存，同一角色每条消息都重新读存储 + PIL 压缩，开销大。用 `avatar_cache`（persona_id → data URL）在一次快照构建内缓存，同角色只压一次。

7. **脱敏要彻底**：快照只能含 user/assistant 文本与图片，绝不能把工具调用参数、引用来源、记忆内容等内部 meta 带进公开快照——那是面向公网的，可能泄露知识库内容或内部细节。

## 六、面试问答（八股）

**Q1：为什么用「快照」而不是「实时镜像」分享对话？**

快照式分享的是「生成分享那一刻的对话内容」，冻结进 `snapshot` JSONB。好处有三：① 语义清晰——用户分享出去的就是当时那段对话，原会话后续继续聊、改名、甚至删除都不影响已分享内容；② 公开访问免去实时拉取会话表的鉴权与一致性问题（公开页无登录态，没法安全地实时读私有会话）；③ 可以在创建时一次性把头像、图片转成 data URL 冻结进去，公开页零鉴权渲染。代价是分享后原对话的新内容不会自动出现在链接里，需要重新分享（同会话复用会刷新快照）。

**Q2：公开页不需要登录，怎么保证安全？**

公开路由 `GET /public/shares/{token}` 挂在独立的 `public_router` 上，**不带 `get_current_user` 依赖**。安全不靠登录态，而靠三道校验：① `share_token` 用 `secrets.token_urlsafe(16)` 生成，随机不可猜不可枚举；② `is_active` 必须为 true（取消即拦下）；③ `expire_at` 过期则返回 404 语义错误。同时快照已脱敏（只含 user/assistant 文本与图片，无工具/引用/记忆内部细节），即使链接泄露也不暴露私有系统数据。

**Q3：公开页的头像和图片怎么在没有鉴权的情况下显示？**

这是关键设计点。项目的文件接口 `/files/{key}` 是带鉴权的，公开页（无登录态）访问会被 401。所以分享创建时就用 PIL 把头像、消息图片读出来、压缩、转成 base64 data URL 内联进快照：头像中心裁剪到 96px JPEG、图片等比缩到长边 1024 JPEG。公开页直接 `<img src="data:image/jpeg;base64,...">` 渲染，完全不依赖鉴权文件接口。压缩是为了控制快照 JSONB 体积，且单条消息限 9 张图、逐张失败降级。

**Q4：同一个会话多次点分享会怎样？**

不会生成一堆链接。`create_share` 先 `get_active_by_conversation` 查该会话是否已有有效分享：有就刷新它的快照、标题、过期、头像，URL（token）不变；没有才新建并生成 token。这样一个会话最多一条有效分享，既避免链接泛滥，又能让用户「再分享一次」时拿到最新对话内容。

**Q5：取消分享为什么用软删？**

`revoke` 只把 `is_active` 置 false，不物理删除行。这样保留了分享痕迹，便于在「我的分享」列表里展示历史状态、做审计；公开查看时 `is_active=false` 会被 `get_public` 拦下返回「已取消」。物理删会丢失这些信息，且无必要——一条软删记录成本很低。

**Q6：群聊对话分享和单聊有什么不同？**

数据结构上复用同一张 `conversation_shares` 表与快照机制，差别在快照构建：群聊的 assistant 消息带 `sender_persona_id`，`_build_snapshot` 会额外为每条带上 `sender_name`（从消息 `meta_data.sender_name` 读，以发言当时的名字为准）和 `sender_avatar`（按角色卡取头像 data URL）。公开页据此像微信群一样按发言人区分展示。为避免同一角色反复读存储压缩头像，用 `avatar_cache` 按 persona_id 缓存，一次快照内同角色只压一次。

**Q7：`conversation_id` 为什么不设外键级联删除？**

因为快照要独立于来源会话存活。分享的语义是「那一刻对话的冻结副本」，用户删掉原会话后，已经分享出去的链接仍应可访问。如果给 `conversation_id` 设 `ondelete=CASCADE`，删会话会连带删掉分享快照，链接突然 404，违背设计意图。所以只把 `conversation_id` 当弱引用存着（带索引便于查同会话有效分享），不设强外键约束。

**Q8：开发期迁移链出过什么问题，怎么避免？**

这条特性的迁移链（`d429ecd1d3a9` → `9596cfc27bc1`）开发期一度因 autogenerate 生成空迁移、`down_revision` 串接混乱而出问题，最后手动核对修正。教训是：alembic autogenerate 不能盲信，每次生成后必须打开脚本确认——upgrade/downgrade 是否真有变更（空迁移要删）、`down_revision` 是否正确接在前驱上、加列是否需要回填，确认无误再 `upgrade head`。
