# 多知识库 — 设计与八股（后端）

> V0.0.3 地基特性。把原来「一个用户一个大库」升级为「分库管理 + 按库开关检索范围」：用户可建多个知识库分门别类放资料，每个库可单独决定「是否参与对话检索」；对话时只检索「已启用检索」的库集合。底层在 Elasticsearch 的 chunk 上加 `kb_id` 字段做多库过滤。本篇只讲后端。

---

## 一、功能定位与需求

- **分库管理**：用户可建多个知识库（如「工作资料」「读书笔记」「项目 A」），文档 / 图片归属到某个库（`documents.kb_id` / `images.kb_id`）。
- **默认库**：每个用户有且仅有一个 `is_default=true` 的默认库（不可删），未指定归属或存量资料都落入默认库。
- **检索开关**：每个库有 `chat_enabled` 字段决定是否参与对话检索。默认库默认开、其余库默认关。
- **对话按库集合检索**：对话时取该用户「所有 `chat_enabled=true` 的库 id」，只在这些库范围内做 RAG 检索（`list_chat_enabled_ids`）。
- **ES 多库过滤**：`comet_chunks` 索引每条 chunk 带 `kb_id`（keyword 类型），检索时 `terms` 过滤限定库范围。
- **删库级联清理**：删库物理删除库内全部文档/图片（PG 靠外键 CASCADE、ES chunk 与对象存储文件在 service 层显式清理）。
- **存量平滑迁移**：上线时跑回填脚本，为每个用户建默认库、把无归属资料归入默认库、回填 ES chunk 的 `kb_id`。

## 二、数据模型与迁移

### 2.1 `knowledge_bases` 表（`api/app/models/knowledge_base_model.py`）

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | `UUID` 主键 | |
| `user_id` | `UUID` FK→users，CASCADE，索引 | 多租户隔离 |
| `name` | `String(128)` | 库名 |
| `description` | `String(512)`，可空 | 简介 |
| `icon` | `String(32)`，可空 | emoji 图标 |
| `color` | `String(16)`，可空 | 卡片主题色 |
| `is_default` | `Boolean`，默认 `false`，索引 | 是否默认库（不可删） |
| `chat_enabled` | `Boolean`，默认 `false` | 是否参与对话检索 |
| `created_at` / `updated_at` | `DateTime(tz)` | |

文档/图片计数**不在本表冗余存储**，而是读取时实时 `GROUP BY kb_id` 统计（`KnowledgeBaseRepository.counts`），避免计数漂移（增删资料时忘了同步计数的经典 bug）。

`documents` / `images` 各加 `kb_id`（`UUID`，FK→knowledge_bases，`ondelete=CASCADE`，带索引）。删库时这两张表的记录靠外键级联删除。

### 2.2 迁移串联

- **`8440c3519b07`（多知识库）**：建 `knowledge_bases` 表 + `documents`/`images` 加 `kb_id` 列 + 索引 + 外键（CASCADE）。`kb_id` 加列时 `nullable=True`——存量行先留空，由回填脚本补归属。
- **`f30e6fa335e9`（知识库加 chat_enabled）**：加 `chat_enabled` 非空布尔列。

`chat_enabled` 迁移用了项目统一的 server_default 套路，但默认值**取 true**：

```python
def upgrade() -> None:
    # 存量知识库都是「默认库」，对话检索默认开启 → server_default true。
    # 加列后去掉 server_default，由 ORM 控制新建库的默认值（普通库默认关）。
    op.add_column('knowledge_bases',
        sa.Column('chat_enabled', sa.Boolean(), nullable=False, server_default=sa.true()))
    op.alter_column('knowledge_bases', 'chat_enabled', server_default=None)
```

关键：迁移时刻已存在的库**都是回填脚本建的默认库**，理应参与检索，所以 `server_default=sa.true()` 回填为开；建列后立刻清除 server_default，之后 ORM 层 `default=False` 接管——**新建的普通库默认关检索**。一个列、两种默认值语义（存量开 / 新建关），靠「server_default 回填后清除」这一手法实现。

## 三、核心实现与代码路径

分层：`knowledge_base_controller.py`（路由）→ `knowledge_base_service.py`（CRUD + 默认库保障 + 删库级联）→ `knowledge_base_repository.py`（数据访问）。ES 侧在 `core/rag/es_index.py`（索引 + 自检）/ `es_store.py`（写入/回填）/ `search.py`（检索过滤）。

### 3.1 默认库保障：读时确保存在

> 文件：`knowledge_base_repository.py` 的 `ensure_default` + `knowledge_base_service.py` 的 `list_kbs`

不靠注册时建默认库，而是**列表接口每次确保默认库存在**（幂等）：

```python
async def list_kbs(self, user_id):
    await self.repo.ensure_default(user_id)   # 没有则建一个默认库
    kbs = await self.repo.list_by_user(user_id)
    ...
```

`ensure_default` 先查 `is_default=true` 的库，没有才建（名「默认知识库」、`is_default=True`、`chat_enabled=True`）。这样老用户、新用户、回填遗漏的用户进入知识库页都能保证有默认库，不依赖某个一次性脚本跑没跑过。

### 3.2 对话检索范围：已启用库集合

> 文件：`knowledge_base_repository.py` 的 `list_chat_enabled_ids`

```python
async def list_chat_enabled_ids(self, user_id):
    stmt = select(KnowledgeBase.id).where(
        KnowledgeBase.user_id == user_id,
        KnowledgeBase.chat_enabled.is_(True),
    )
    rows = (await self.session.execute(stmt)).scalars().all()
    return [str(r) for r in rows]
```

对话时 `chat_service._build_tools` 调它拿到「已启用检索的库 id 列表」，传给检索工具。返回的是字符串 id 列表（与 ES `kb_id`（keyword）对齐）。

### 3.3 ES 多库过滤

> 文件：`core/rag/es_index.py`（mapping）+ `es_store.py`（写入）+ `search.py`（过滤）

mapping 里 `kb_id` 是 **keyword** 类型（精确匹配、可 `terms` 过滤）：

```python
"kb_id": {"type": "keyword"},  # 所属知识库（多知识库检索范围过滤）
```

写入时 `build_chunk_doc` 带上 `kb_id`。检索时 `hybrid_search` 把 `kb_ids` 加进 `base_filter`，向量召回和 BM25 召回都用同一个 filter：

```python
if kb_ids is not None:
    base_filter.append({"terms": {"kb_id": kb_ids}})
```

语义约定（重要）：

- `kb_ids=None` → **不限库**（全库检索，如知识库页内的全局语义检索）。
- `kb_ids=[]`（空列表）→ **没有任何启用的库**，`terms` 匹配空集 → 直接返回空（对话时一个库都没开就不检索）。
- `kb_ids=[id1, id2]` → 只检索这些库。

### 3.4 **关键坑与修复**：旧索引 `kb_id` 被动态映射成 text

> 文件：`core/rag/es_index.py` 的 `ensure_index` / `_kb_id_type` / `_rebuild_index_fix_kb_id`

这是多知识库最大的实现陷阱。`comet_chunks` 索引在多知识库上线**之前**就存在了，那时没有 `kb_id` 字段。回填脚本往老 chunk 里写 `kb_id` 时，ES 的**动态映射**会按值自动推断类型，把 `kb_id` 推断成 `text`（带 `.keyword` 子字段）而非纯 `keyword`。而 `text` 类型经过分词，**无法用 `terms` 做精确多库过滤**——多库过滤直接失效。

更糟的是：**ES 不允许原地修改已存在字段的类型**（`text`→`keyword` 改不了）。所以只能重建索引。

解决方案是把 `ensure_index` 从「不存在才建」改成**启动自检 + 必要时自动 reindex 重建**：

```python
async def ensure_index() -> None:
    exists = await es.indices.exists(index=CHUNKS_INDEX)
    if not exists:
        await es.indices.create(index=CHUNKS_INDEX, body=_MAPPING)  # 全新：直接按正确 mapping 建
        return
    kb_type = await _kb_id_type(es)          # 读当前 kb_id 字段类型
    if kb_type == "keyword":
        return                                # 已正确，无需处理
    if kb_type is None:
        await es.indices.put_mapping(...)     # 字段缺失：增量补（不影响存量）
        return
    # 类型不对（多为 text）：reindex 重建
    await _rebuild_index_fix_kb_id(es)
```

`_rebuild_index_fix_kb_id` 用**临时索引中转**重建，保证幂等、不丢数据：

```
建临时索引(正确 mapping)
  → reindex 原 → 临时 (refresh=True 等数据落盘)
  → 删原索引，按正确 mapping 重建
  → reindex 临时 → 原
  → 删临时索引
```

进入时先清理可能残留的临时索引（上次中断遗留），保证可重复执行。数据量小（个人项目），串行同步 `wait_for_completion=True` 等待即可。这一步跑完，`kb_id` 才真正是 keyword，多库过滤生效。

### 3.5 存量回填脚本

> 文件：`api/app/db/backfill_kb.py`（`uv run python -m app.db.backfill_kb`）

上线时一次性迁移存量数据，三件事 + 幂等：

```python
for uid in user_ids:
    kb = await kb_repo.ensure_default(uid)                 # 1) 建/取默认库
    await session.execute(update(Document).where(          # 2) 无归属文档归默认库
        Document.user_id == uid, Document.kb_id.is_(None)).values(kb_id=kb.id))
    await session.execute(update(Image).where(             #    无归属图片归默认库
        Image.user_id == uid, Image.kb_id.is_(None)).values(kb_id=kb.id))
    await session.commit()
    es_updated = await backfill_kb_id(str(uid), str(kb.id)) # 3) ES chunk 回填 kb_id
```

`backfill_kb_id`（`es_store.py`）用 `update_by_query`，只更新「该用户、缺失 kb_id 字段」的 chunk（`must_not exists kb_id`），把它们的 `kb_id` 写成默认库 id。幂等性：已有默认库则复用、已归属资料 `kb_id.is_(None)` 过滤跳过、ES 仅回填缺 `kb_id` 的 chunk。脚本用任务级独立引擎（`create_task_engine`），跑完 dispose + 关 ES 连接。

> 注意启动顺序：`ensure_index` 的 reindex 修复必须在回填之后能保证类型正确——实际上无论回填先把 `kb_id` 写成 text，下次启动 `ensure_index` 自检都会 reindex 修成 keyword。两者配合，最终一致。

### 3.6 删库级联清理

> 文件：`knowledge_base_service.py` 的 `delete`

删库是物理删除，分两步：

```python
if kb.is_default:
    raise BizError("默认知识库不可删除", code=3041, status_code=400)
# 1) 清理库内文档/图片的 ES chunk + 对象存储文件（DB 不感知这两处）
for d in docs:
    await delete_by_source(str(user_id), str(d.id)); await storage.delete(d.file_key)
for im in imgs:
    await delete_by_source(str(user_id), str(im.id)); await storage.delete(im.file_key)
# 2) 删库（外键 CASCADE 级联删 documents/images 的 PG 记录）
await self.repo.delete(kb)
```

PG 记录靠外键 `ondelete=CASCADE` 自动删，但**ES 向量 chunk 和对象存储文件 DB 感知不到**，必须 service 层显式清理。每个文档/图片的清理包 try/except，单个失败只 warning 跳过，不阻断整库删除。默认库受保护不可删。

## 四、设计取舍（已定决策）

| 决策 | 选择 | 理由 |
|------|------|------|
| 分库实现 | `documents`/`images` 加 `kb_id` 外键，不拆多索引 | 个人版数据量小，单 ES 索引 + `kb_id` 过滤足够；不必为每库建独立索引 |
| 计数存储 | 读时实时 `GROUP BY` 统计，不冗余字段 | 避免「增删资料忘了同步计数」的漂移 bug |
| 默认库 | 每用户一个、不可删、读时 `ensure_default` 保障 | 给无归属资料一个兜底归属；幂等保障不依赖一次性脚本 |
| 检索范围 | 取 `chat_enabled=true` 的库集合 | 用户可精确控制「哪些库参与对话」，避免无关库污染检索 |
| `chat_enabled` 默认 | 默认库开、普通库关（迁移 server_default true 后清除） | 默认库是兜底主库该参与；新建专题库默认不打扰对话 |
| ES `kb_id` 类型 | keyword + 启动自检 reindex 修复 | keyword 才能精确 `terms` 过滤；旧索引被推断成 text 无法原地改，只能重建 |
| `kb_ids` 语义 | None=不限 / []=空集不检索 / [..]=限定 | 区分「全库检索」与「没开任何库」两种场景 |
| 删库 | 物理删除 + 二次确认（前端） | 个人项目，物理删够用；ES/OSS 显式清理 |
| 存量迁移 | 独立幂等回填脚本 | 一次性迁移可重复执行，不混进自动迁移 |

## 五、易踩坑点

1. **ES 动态映射把 `kb_id` 推断成 text**：这是核心坑。老索引没有 `kb_id`，回填写入时 ES 动态映射按值推断成 `text`，导致 `terms` 多库过滤失效。且 ES 不能原地改字段类型，只能 reindex 重建。`ensure_index` 启动自检 + 临时索引中转重建解决，幂等可重复。

2. **`kb_ids=[]` 与 `kb_ids=None` 含义完全不同**：空列表是「没有任何启用的库 → 不检索、返回空」；`None` 是「不限库 → 全库检索」。调用方一定要分清：对话场景传库集合（可能是空列表），知识库页全局检索传 `None`。

3. **删库要显式清 ES + OSS**：PG 靠外键 CASCADE 删记录，但 ES 向量 chunk 和对象存储文件 DB 感知不到，必须在 service 层 `delete_by_source` + `storage.delete`。漏了会留下孤儿 chunk（检索还能搜到已删资料）和孤儿文件（占存储）。

4. **默认库保护**：删库前必须判 `is_default` 拦截，否则删了默认库会导致无归属资料无处可去、`ensure_default` 又会重建一个空库。

5. **回填脚本要 dispose 引擎 + 关 ES**：脚本用任务级独立引擎，跑完必须 `engine.dispose()` + `elastic.close()`，否则连接泄漏、脚本不退出。

6. **计数实时统计的代价**：`counts` 每次列表都 `GROUP BY` 两次（文档 + 图片）。数据量大时可考虑缓存，但个人项目量小，实时统计换「绝不漂移」是划算的。

7. **`chat_enabled` 一列两默认**：迁移 server_default=true 是给**存量**用的，建列后立即清除，新建库走 ORM `default=False`。如果不清除 server_default，新建普通库会被数据库默认成 true，与「普通库默认关」矛盾。

## 六、面试问答（八股）

**Q1：多知识库为什么不给每个库建一个独立 ES 索引，而是共用一个索引加 `kb_id` 过滤？**

个人项目数据量小，单索引 `comet_chunks` + `user_id`/`kb_id` 过滤完全够用，且运维简单（一个索引、一套 mapping、一次 reindex）。每库一个索引会带来索引数量爆炸、跨库检索要 multi-index 查询、mapping 维护成本高等问题，是企业级大数据量才需要的方案。我们用 `kb_id`（keyword）做 `terms` 过滤，在召回阶段就限定库范围，向量召回和 BM25 召回共用同一个 filter，实现简单且性能足够。

**Q2：`kb_id` 为什么必须是 keyword 而不是 text？踩了什么坑？**

`terms` 精确过滤要求字段是 keyword（不分词、精确匹配）。`text` 会被分词，无法精确匹配整个 id，多库过滤就失效了。踩的坑是：`comet_chunks` 索引在多知识库上线前就存在、没有 `kb_id` 字段，回填写入时 ES 的**动态映射**按值自动把 `kb_id` 推断成 `text`。而 ES **不允许原地修改已有字段类型**。所以我把 `ensure_index` 改成启动自检：读 `kb_id` 当前类型，是 keyword 就跳过、缺失就 `put_mapping` 补、是 text 就**自动 reindex 重建**（临时索引中转、幂等）。重建后 `kb_id` 才真正是 keyword，多库过滤生效。

**Q3：reindex 重建的具体流程？怎么保证不丢数据、可重复执行？**

流程是临时索引中转：① 建临时索引（正确 mapping）；② reindex 原→临时，`refresh=True` 确保数据落盘后再删原索引；③ 删原索引、按正确 mapping 重建；④ reindex 临时→原；⑤ 删临时索引。不丢数据靠「先把数据搬到临时索引并落盘，再删原索引」的顺序保证。可重复执行靠进入时先清理可能残留的临时索引（上次中断遗留），且 `ensure_index` 每次启动都自检——只要 `kb_id` 不是 keyword 就会再修，修成 keyword 后自检直接跳过。数据量小，串行同步 `wait_for_completion=True` 等待即可。

**Q4：对话时怎么决定检索哪些库？`kb_ids` 几种取值含义？**

对话时取该用户「所有 `chat_enabled=true` 的库 id」（`list_chat_enabled_ids`），作为 `kb_ids` 传给 `hybrid_search`，只在这些库里检索。`kb_ids` 三种语义：`None`=不限库（全库检索，用于知识库页内的全局语义检索）；`[]` 空列表=没有任何启用的库，`terms` 匹配空集直接返回空（用户一个库都没开就不检索）；`[id1,id2]`=只检索指定库。这样用户能通过 `chat_enabled` 开关精确控制「哪些库参与对话」，避免无关库污染回答。技能绑库时还能进一步把范围限定到单个库（优先级更高）。

**Q5：库的文档/图片计数为什么不存字段，每次实时算不慢吗？**

不存冗余计数是为了**避免漂移**——一旦存字段，每次增删文档/图片都要记得同步更新计数，任何一处遗漏（异步任务失败、并发、异常路径）都会导致计数与实际不符，这类 bug 很隐蔽。所以改成读取时实时 `GROUP BY kb_id` 统计（文档、图片各一次聚合）。个人项目数据量小，两次聚合查询开销可忽略，用一点查询成本换「计数永远准确」很划算。数据量真的大了再加缓存层也不迟。

**Q6：删一个知识库时，数据是怎么被清理干净的？**

三处数据要清：PG 记录、ES 向量 chunk、对象存储文件。PG 的 `documents`/`images` 记录靠 `kb_id` 外键 `ondelete=CASCADE` 自动级联删除。但 ES chunk 和 OSS 文件**数据库感知不到**，必须在 `KnowledgeBaseService.delete` 里显式遍历库内文档/图片，对每个调 `delete_by_source`（删 ES chunk）+ `storage.delete`（删文件），单个失败只 warning 跳过不阻断。最后删库行触发 PG 级联。另外默认库受保护，删前判 `is_default` 拦截。漏清 ES/OSS 会留下「已删资料还能被检索到」的孤儿 chunk 和占空间的孤儿文件。

**Q7：存量数据怎么平滑迁移到多知识库？为什么用单独脚本而不是放进 Alembic 迁移？**

用独立回填脚本 `backfill_kb.py`：为每个用户 `ensure_default` 建默认库，把 `kb_id IS NULL` 的文档/图片 `UPDATE` 归入默认库，再用 ES `update_by_query` 把缺 `kb_id` 的 chunk 回填成默认库 id。它是幂等的（已有默认库复用、已归属资料跳过、ES 只补缺失的），可重复执行。不放进 Alembic 是因为：Alembic 迁移应聚焦 schema 变更（DDL），而这是**数据迁移**且涉及 ES（Alembic 只管 PG），混进去会让迁移变重、不可重试、ES 失败还会卡住 schema 升级。拆成独立脚本，schema 迁移和数据回填解耦，各自可控。

**Q8：`chat_enabled` 这个布尔列在迁移里为什么 server_default 是 true，但 ORM 默认是 false？**

因为「存量库」和「新建库」要不同默认值。迁移执行时刻数据库里已有的库都是回填脚本建的**默认库**，默认库理应参与对话检索，所以加列时 `server_default=sa.true()` 把存量行回填为开。但建列后立刻 `alter_column(server_default=None)` 清除数据库默认，之后新行的默认值交给 ORM 层 `default=False` 控制——用户**新建的普通专题库默认关闭检索**，不主动打扰对话。一个列通过「server_default 回填后清除」实现了「存量开、新建关」两种默认语义，同时保持数据库 schema 干净（不残留 server_default）。
