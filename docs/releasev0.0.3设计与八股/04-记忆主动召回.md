# 记忆主动召回 — 设计与八股（后端）

> V0.0.3 记忆增强特性。让 AI「不用查就先知道」：每轮对话开始前，用当前问题去检索相关的记忆事实（实体 + 关系）和高层洞察（Insight），拼成一段「关于用户的已知信息」背景块，追加到 system prompt。模型回答时自然带着对用户的了解，而不必每次靠记忆工具临时查。本篇只讲后端。

---

## 一、功能定位与需求

记忆系统原本是「按需查询」的：把记忆检索包成一个 LangChain 工具，由 LLM 自主决定要不要调。但这有两个问题：① 弱模型/不爱调工具的模型经常该查不查，对话显得「不记得用户」；② 即使调了，也是一次额外的工具往返，增加延迟。主动召回补的是「主动提供基础背景」这一层：

- **每轮主动召回**：对话每轮用当前问题召回**洞察 top-2 + 实体事实 top-5**，拼成背景块注入 system prompt，模型开口前就「知道」相关背景。
- **余弦门控节流**：实体召回带余弦相似度门控（阈值 0.5），相关度不够的不注入，避免无关记忆挤占上下文、带偏回答。
- **性能优先**：召回是锦上添花，绝不能拖累对话首字延迟——query 只 embed 一次复用、洞察与实体两路并行 gather、整体 3.5s 超时，超时直接放弃注入。
- **开关可控**：`agent_configs.enable_active_recall` 默认开，用户可在角色配置页关闭。
- **与记忆工具并存**：主动召回给「稳定基础背景」，记忆工具供 LLM「主动查更细」，两者互补不冲突。

## 二、数据模型与迁移

主动召回本身不新建表，只在 `agent_configs` 加一个开关字段，召回数据来自既有的 Neo4j 记忆图谱（实体/关系/洞察）。

### 2.1 开关字段（`api/app/models/agent_config_model.py`）

`AgentConfig` 加一列：

| 字段 | 类型 | 默认 | 说明 |
|------|------|------|------|
| `enable_active_recall` | `Boolean` | `True` | 每轮提问是否自动召回相关记忆 + 洞察注入上下文 |

### 2.2 迁移（`api/app/migrations/versions/cf7ae75691ea_agent_config_add_enable_active_recall.py`）

非空布尔列加列，沿用「先 `server_default` 回填存量行，再清除默认」的套路——默认开启（`server_default=sa.true()`），让存量用户也享受主动召回：

```python
op.add_column(
    'agent_configs',
    sa.Column('enable_active_recall', sa.Boolean(), nullable=False, server_default=sa.true()),
)
op.alter_column('agent_configs', 'enable_active_recall', server_default=None)
```

先用 `server_default=sa.true()` 给所有存量行回填 `true`，加列成功后立刻清除默认——之后新行的默认值由 ORM 层（`default=True`）控制，保持数据库 schema 干净。

## 三、核心实现与代码路径

分层：`chat_service.py`（调用点 + 开关判断）→ `core/memory/retrieval/active_recall.py`（召回编排 + 超时 + 拼块）→ `core/memory/retrieval/searcher.py`（实体混合检索）+ `memory_graph_repository.search_insights_by_vector`（洞察向量召回）。

### 3.1 调用点（`api/app/services/chat_service.py`）

`stream_chat` 组装 system prompt 时，按开关决定是否召回：

```python
history = await self._history_messages(conv.id)
# 主动记忆召回：每轮用当前问题检索相关记忆 + 洞察，注入 system prompt（开关控制）
if agent is None or agent.enable_active_recall:
    recall = await self._recall_memory(user_id, user_text)
    if recall:
        system_prompt = (system_prompt + "\n\n" + recall).strip()
```

`_recall_memory` 是一层薄封装：拿用户的 embedding 客户端（没配则返回空串），调 `recall_context`，整段 try/except 包住——**召回失败绝不影响对话**：

```python
async def _recall_memory(self, user_id, query) -> str:
    try:
        embed_client = await get_optional_client_for_type(self.session, user_id, "embedding")
        if embed_client is None:
            return ""
        return await recall_context(embed_client=embed_client, user_id=user_id, query=query)
    except Exception:
        return ""
```

注意 `agent is None or agent.enable_active_recall`：用户还没建过 agent 配置时（agent 为 None）也默认召回，与字段默认值 `True` 行为一致。

### 3.2 召回编排（`api/app/core/memory/retrieval/active_recall.py`）

`recall_context` 外层做超时与容错；`_do_recall` 做实际召回：

**性能优化一：query 只 embed 一次。** 对话首字延迟敏感，embedding 是网络调用，不能算两次：

```python
qvec = await embed_client.embed_one(query)   # 只算一次
```

这个向量同时给洞察召回（`search_insights_by_vector(uid, qvec, ...)`）和实体召回（`search_memory(..., query_vector=qvec)` 通过 `query_vector` 参数复用），避免重复 embedding。

**性能优化二：两路并行。** 洞察召回和实体召回互不依赖，用 `asyncio.gather` 并发：

```python
insight_lines, memory_lines = await asyncio.gather(
    _recall_insights(), _recall_entities()
)
```

**性能优化三：整体超时。** 召回是锦上添花，慢了就放弃，不拖累对话：

```python
_RECALL_TIMEOUT = 3.5
try:
    return await asyncio.wait_for(_do_recall(...), timeout=_RECALL_TIMEOUT)
except asyncio.TimeoutError:
    return ""   # 超时直接让模型回答
```

每个子召回（洞察/实体/向量化）也各自 try/except 单独降级，任一路失败只 warning、返回空，不影响另一路。

### 3.3 余弦门控节流

实体召回传 `min_vector_score=settings.active_recall_min_score`（0.5），`searcher.search_memory` 据此启用「绝对相关度门控」——只保留向量余弦 ≥ 0.5 的实体。这是为了**节流防噪声**：召回的本质是「跟当前问题真正相关的才注入」，如果不门控，向量检索总会返回 top-k 最近邻（哪怕完全不相关），把无关记忆塞进 prompt 反而带偏模型。门控让「没相关记忆时干脆不注入」。

洞察召回数量少（top-2）、本身是高度概括的画像结论，相关性靠向量索引排序保证，注入价值高。

### 3.4 背景块拼装

命中后拼成结构化背景块，明确告诉模型「这是参考信息，可自然融入、不必刻意提及」：

```
【关于用户的已知信息（供参考，可自然融入回答，不必刻意提及）】
我对用户的理解：是一位持续精进的后端工程师；倾向于自驱学习新技术
相关记忆：
- 多多：用户养的小狗
  · 用户 拥有 多多
- Rust：一门系统编程语言
```

洞察用「我对用户的理解：」前缀、分号连接；实体事实用「相关记忆：」列表，每个实体带描述 + 最多 2 条一跳关系。最后做长度兜底：超过 `active_recall_max_chars`（600）截断加省略号，防止背景块过长挤占对话上下文。

两路都空则返回空串（不注入空块）。

### 3.5 配置项（`api/app/config.py`）

```python
active_recall_entity_top_k: int = 5    # 召回实体数
active_recall_insight_top_k: int = 2   # 召回洞察数
active_recall_min_score: float = 0.5   # 实体召回余弦门控（低于不注入，节流防噪声）
active_recall_max_chars: int = 600     # 注入背景块长度上限
```

## 四、设计取舍（已定决策）

| 决策 | 选择 | 理由 |
|------|------|------|
| 召回 vs 工具 | 主动召回 **+** 记忆工具并存 | 召回给稳定基础背景（解决弱模型不调工具），工具供 LLM 查更细，互补 |
| 召回内容 | 洞察 top-2 + 实体事实 top-5 | 洞察给画像级理解，实体给具体事实，量小精准 |
| 节流 | 实体召回余弦门控 0.5 | 没相关记忆就不注入，避免无关噪声带偏回答 |
| embedding | query 只算一次，两路复用 | 省一次网络调用，降首字延迟 |
| 并发 | 洞察/实体两路 `gather` 并行 | 两路无依赖，并行省时 |
| 超时 | 整体 3.5s，超时放弃注入 | 召回是锦上添花，绝不拖累对话首字延迟 |
| 开关 | `enable_active_recall` 默认开 | 主动记忆是正向体验，默认开；用户可关 |
| 失败处理 | 全链路 try/except，失败返回空串 | 召回是副作用，绝不影响主对话 |
| 注入方式 | 追加 system prompt 背景块 | 让模型「开口前就知道」，比插历史消息更自然 |

## 五、易踩坑点

1. **AsyncSession 非并发安全，召回的并发只在召回内部**：chat_service 里主动召回、工具构建、跨会话上下文三步共用同一个 `self.session`，不能并发（SQLAlchemy AsyncSession 非并发安全），所以是串行的。召回内部的并行（`gather` 两路）走的是 Neo4j 驱动和独立的 embedding HTTP 调用，不碰那个 PG session，所以可以并发。两者别混淆。

2. **query 向量复用要靠参数显式传入**：`search_memory` 默认会自己 `embed_one`，主动召回必须显式传 `query_vector=qvec` 才会复用，否则又多算一次 embedding。这是 searcher 专门留的复用口子。

3. **门控阈值是「相关度」不是「召回数量」**：`min_vector_score=0.5` 意味着「不够相关就一个都不注入」，是有意为之的节流。如果调太低会把无关记忆塞进 prompt 带偏回答，调太高会经常召不回任何东西。0.5 是实测平衡点。

4. **超时要包整个召回，不是单路**：`asyncio.wait_for` 包的是 `_do_recall` 整体（含 embedding + 两路 gather），任一环节卡住，整体 3.5s 到点就放弃。别只给某一路加超时，否则另一路卡住照样拖。

5. **背景块措辞要弱化「指令感」**：注入的是「供参考、可自然融入、不必刻意提及」，避免模型每句话都生硬地复述「根据我对你的了解…」。措辞直接影响体验。

6. **agent 为 None 时也要召回**：用户没建过配置时 agent 是 None，判断写成 `agent is None or agent.enable_active_recall`，保证新用户也默认享受召回，与字段默认 `True` 一致。

7. **召回不做记忆萃取**：主动召回只「读」记忆图谱，对话回答后的「写记忆」（萃取）是另一条异步链路（`_dispatch_memory`），两者独立。

## 六、面试问答（八股）

**Q1：已经有「记忆检索工具」了，为什么还要做主动召回？两者冲突吗？**

不冲突，是互补。记忆工具是「按需查询」——包成 LangChain 工具，由 LLM 自主决定调不调。问题是弱模型或不爱调工具的模型经常该查不查，对话显得「不记得用户」，而且每次调都是一次额外往返增加延迟。主动召回是「主动提供基础背景」——每轮对话开始前就用当前问题召回相关洞察和事实，拼进 system prompt，模型开口前就带着对用户的了解。所以主动召回兜底「稳定的基础背景」，记忆工具供模型「想查更细时主动查」，两层并存。

**Q2：主动召回怎么保证不拖累对话首字延迟？**

三个优化叠加。① query 只 embed 一次：embedding 是网络调用，召回的洞察路和实体路复用同一个查询向量，靠 `search_memory` 的 `query_vector` 参数显式传入避免重算。② 两路并行：洞察召回和实体召回互不依赖，用 `asyncio.gather` 并发跑。③ 整体 3.5s 超时：用 `asyncio.wait_for` 包住整个召回，超时直接返回空串、放弃注入让模型先答。召回本来就是锦上添花，宁可不注入也不能卡住对话。再加上全链路 try/except，任何失败都返回空串。

**Q3：召回回来的记忆怎么过滤？为什么要余弦门控？**

实体召回带余弦相似度门控，阈值 0.5，只保留与当前问题向量余弦 ≥ 0.5 的实体。原因是向量检索（kNN）总会返回 top-k 最近邻，哪怕这些记忆跟当前问题毫不相关。如果不门控，无关记忆被塞进 prompt 会带偏模型回答、还挤占上下文。门控的语义是「没有足够相关的记忆，就干脆一个都不注入」。这是节流，不是限量。

**Q4：召回了什么内容？怎么注入？**

召回洞察 top-2（反思引擎归纳的画像级理解）+ 实体事实 top-5（实体 + 描述 + 最多 2 条一跳关系）。拼成一段「关于用户的已知信息」背景块，追加到 system prompt 末尾。背景块里明确标注「供参考、可自然融入回答、不必刻意提及」，避免模型生硬复述。还做了长度兜底（超 600 字截断）。注入 system prompt 而非插历史消息，是因为这是「模型应当知道的背景」，放 system 里最自然。

**Q5：开关是怎么控制的？默认值为什么是开？**

`agent_configs.enable_active_recall`，默认 `True`。判断写成 `agent is None or agent.enable_active_recall`——用户还没建配置（agent 为 None）时也默认召回，和字段默认值一致。默认开是因为主动记忆是正向体验（AI 记得你）。迁移加列时用 `server_default=sa.true()` 让存量用户也默认开启，加完列再清除 server_default 由 ORM 层控制。

**Q6：主动召回和反思引擎、记忆萃取是什么关系？**

数据流上：记忆萃取（写）把对话沉淀成实体/陈述存进 Neo4j；反思引擎（归纳）回看高频高重要实体产出洞察并向量化；主动召回（读）每轮用问题向量去检索实体事实和洞察，注入对话。召回是纯「读」侧消费者，消费萃取和反思的产出。召回本身不写记忆——对话回答后的萃取是另一条独立的异步链路。

**Q7：召回失败了对话会怎样？**

完全无感。`_recall_memory` 整段 try/except 返回空串；`recall_context` 外层 try/except + 超时返回空串；内部每路召回（向量化/洞察/实体）各自 try/except 单独降级。任何一环失败或超时，最终就是「不注入背景块」，模型照常基于 system prompt + 历史 + 当前问题回答。这是「副作用失败不阻断主流程」原则的典型应用——召回是增强，不是对话的必要前提。

**Q8：为什么洞察召回不做门控，实体召回要做？**

洞察召回数量很少（top-2），且洞察本身是高度概括的画像结论（「是一位持续精进的后端工程师」），相关性靠向量索引的排序保证，注入价值天然高、噪声风险低。实体事实数量多（top-5）、颗粒细，没门控容易混入完全不相关的具体事实（比如问技术问题却召回「养了只狗」），所以实体路才需要余弦门控把不相关的挡掉。这是按两类数据的特性做的差异化处理。
