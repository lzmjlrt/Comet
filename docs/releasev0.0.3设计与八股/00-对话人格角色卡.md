# 对话人格（角色卡）— 设计与八股（后端）

> V0.0.3 第一批特性。让用户为同一个助手维护多组「人设」（角色卡），随时一键切换当前生效的人格，对话时把该人格的 `system_prompt` 与 `temperature` 注入模型，实现「换个角色聊天」。本篇只讲后端。

---

## 一、功能定位与需求

把原本固定在 `agent_configs` 里的「一个全局人设」，升级为「多组可切换人设」：

- **多组人格并存**：一个用户可以建多张角色卡（如「严谨助理」「周杰伦」「冷面吐槽役」），每张卡有自己的组名、头像、人格提示词、采样温度。
- **互斥生效**：同一时刻只有一张卡 `is_active=true`（当前生效）。切换是「先全部置 false，再激活目标」的原子操作。
- **对话注入**：发消息时取当前生效人格，把它的 `system_prompt` 作为 system message 注入、`temperature` 作为采样温度传给模型。
- **头像存 key 不存 URL**：角色头像在库里存对象存储的 `avatar_key`（如 `avatars/xxx.png`），出参时再用存储后端临时换成可访问 URL；URL 会过期、迁移存储后会变，存 key 才稳定。
- **用户级头像开关**：`agent_configs.show_avatar` 控制对话界面是否显示头像（开 → 显示 AI 人格头像 + 用户头像；关 → 两边都不显示）。这是「用户级」开关，不是「人格级」。

与 `agent_configs`（每用户一条的全局 Agent 配置：工具开关、主动召回、跨会话等）的分工：**人格卡只管「我是谁」（人设 + 语气 + 温度 + 头像）**，工具/记忆/召回等行为开关仍在 `agent_configs`。两者在对话时叠加。

## 二、数据模型与迁移

### 2.1 `agent_personas` 表（`api/app/models/agent_persona_model.py`）

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | `UUID` 主键 | |
| `user_id` | `UUID` FK→users，`ondelete=CASCADE`，带索引 | 多租户隔离，删用户连带删人格 |
| `name` | `String(64)` | 组名（如「周杰伦」「严谨助理」） |
| `avatar_key` | `String(512)`，可空 | 头像**对象存储 key**（非 URL）；为空则该人格不显示头像 |
| `system_prompt` | `Text`，默认 `""` | 人格提示词（人设/语气/口头禅），对话时作为 system message 注入 |
| `temperature` | `Float`，默认 `0.7` | 该人格的采样温度 |
| `is_active` | `Boolean`，默认 `false`，带索引 | 是否当前生效，每用户最多一条 `true`（service 层保证互斥） |
| `sort` | `Integer`，默认 `0` | 列表排序（预留） |
| `created_at` / `updated_at` | `DateTime(tz)` | |

### 2.2 `agent_configs.show_avatar`（`api/app/models/agent_config_model.py`）

头像开关挂在用户级全局配置 `agent_configs` 上，而不是每张角色卡上：

```python
# 对话界面是否显示头像（开 → AI 人格头像 + 用户头像；关 → 两边都不显示）
show_avatar: Mapped[bool] = mapped_column(Boolean, default=False)
```

放这里的理由：头像显隐是「界面偏好」，对所有人格统一生效，不该让用户给每张卡分别配；而 `agent_configs` 本就是「每用户一条」的全局配置载体，天然合适。

### 2.3 迁移要点

- `agent_personas` 建表迁移随表创建（id/user_id/name/avatar_key/system_prompt/temperature/is_active/sort + 两个时间戳 + user_id、is_active 索引 + user_id 外键 CASCADE）。
- `show_avatar` 作为非空布尔列加到 `agent_configs`：遵循项目统一手法——先 `server_default=sa.false()` 让存量行回填，再 `alter_column(server_default=None)` 清除默认，之后新行默认值由 ORM 层 `default=False` 控制，保持数据库 schema 干净。

> 没有给 `agent_personas` 建「每用户唯一 active」的数据库部分唯一索引，互斥靠 service 层事务保证（见 §三）。这是有意取舍：部分唯一索引（`WHERE is_active`）在切换的中间状态（短暂出现两条都将被更新）下不好处理，应用层串行事务更简单可控。

## 三、核心实现与代码路径

分层：`agent_persona_controller.py`（路由）→ `agent_persona_service.py`（业务 + 互斥事务 + 出参组装）→ `agent_persona_repository.py`（数据访问，全部带 user_id 隔离）。对话注入在 `chat_service.py`。

### 3.1 互斥激活：先全灭再点亮（单事务）

> 文件：`agent_persona_service.py` 的 `activate` + `agent_persona_repository.py` 的 `deactivate_all`

切换当前生效人格必须保证「全局只有一条 `is_active=true`」。实现是：

```python
async def activate(self, user_id, persona_id):
    persona = await self._get_or_404(user_id, persona_id)
    await self.repo.deactivate_all(user_id)   # 不提交：UPDATE ... SET is_active=false WHERE user_id=? AND is_active
    persona.is_active = True
    saved = await self.repo.save(persona)      # 这里 commit：批量灭 + 单条亮 一起提交
    return saved
```

关键点：

- `deactivate_all` 只发 `UPDATE` **不提交**（`session.execute(update(...))`），把「灭掉其余」和「点亮目标」放进**同一个事务**，由 `repo.save` 的 `commit` 一起落库。中途异常则整体回滚，不会留下「零条生效」或「两条生效」的脏状态。
- `deactivate_all` 的 `WHERE` 带 `is_active.is_(True)`，只更新当前亮着的那条，不全表扫。

### 3.2 首张卡自动生效

> 文件：`agent_persona_service.py` 的 `create`

新建人格时，如果这是该用户的**第一张**卡，自动设为 `is_active=true`，避免「建了卡但没有任何生效人格」的空窗：

```python
if await self.repo.count(user_id) == 0:
    persona.is_active = True
```

### 3.3 头像 key → URL 的出参组装

> 文件：`agent_persona_service.py` 的 `to_out_dict`

库里存的是 `avatar_key`，前端要的是能直接 `<img>` 的 URL，所以出参时调存储后端换 URL，且**失败只 warning 不阻断**列表返回：

```python
avatar_url = None
if persona.avatar_key:
    try:
        avatar_url = get_storage().get_url(persona.avatar_key)
    except Exception as e:  # 取 url 失败不影响列表返回
        logger.warning("角色头像 url 生成失败: key=%s err=%s", persona.avatar_key, e)
```

出参同时返回 `avatar_key`（前端编辑回填用）和 `avatar_url`（展示用）。

### 3.4 对话注入：取生效人格的 prompt + temperature

> 文件：`api/app/services/chat_service.py` 的 `stream_chat` + `_compose_system_prompt`

发消息时：

```python
persona = await self.persona_repo.get_active(user_id)
temperature = persona.temperature if persona else 0.7
...
model, config = await build_default_chat_model(self.session, user_id, temperature=temperature, streaming=True)
system_prompt = self._compose_system_prompt(persona, skill)
```

- **温度**：取生效人格的 `temperature`，没有生效人格时兜底 `0.7`。温度在构建模型时就传入（`build_default_chat_model(..., temperature=...)`），决定本轮采样。
- **system prompt**：`_compose_system_prompt(persona, skill)` 把**人格人设**放在最前，技能任务提示词叠加在后（技能见 `02-Skills技能.md`）。人格定「我是谁」，技能叠加「我现在干什么」：

```python
parts: list[str] = []
persona_prompt = (persona.system_prompt.strip() if persona else "") or ""
if persona_prompt:
    parts.append(persona_prompt)
if skill:
    ...  # 追加 【当前任务能力：xxx】 + few-shot
return "\n\n".join(parts)
```

注入后，`chat_service` 还会在 system prompt 末尾追加「当前日期 + 时效引导」块（`context_hint`），并按开关追加主动召回 / 跨会话上下文。所以最终 system prompt = 人格人设 + 技能 + 日期块 + （可选）召回 + （可选）跨会话。

### 3.5 接口（`agent_persona_controller.py`，前缀 `/personas`）

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/personas` | 列出当前用户全部人格（含 avatar_url、is_active） |
| POST | `/personas` | 新建人格（首张自动生效） |
| PUT | `/personas/{id}` | 编辑（传啥改啥；`avatar_key` 传空串表示移除头像） |
| DELETE | `/personas/{id}` | 删除 |
| POST | `/personas/{id}/activate` | 设为当前生效（互斥切换） |

`PersonaUpdate` 用 `model_dump(exclude_unset=True)` 实现「传啥改啥」的部分更新；`avatar_key` 字段特殊：传空串/None 显式置 `None`（移除头像），区别于「未传该字段」。

## 四、设计取舍（已定决策）

| 决策 | 选择 | 理由 |
|------|------|------|
| 多人格存储 | 独立 `agent_personas` 表，每用户多行 | 人格是「一对多」的可切换集合，独立表天然支持；不挤进每用户一条的 `agent_configs` |
| 生效互斥 | service 层事务（全灭→点亮），**不建部分唯一索引** | 切换中间态下部分唯一索引不好处理，应用层串行事务更简单可控 |
| 头像存储 | 存 `avatar_key`（对象存储 key），出参换 URL | URL 会过期、换存储会变；存 key 稳定，展示时实时换 URL |
| 头像开关粒度 | 用户级（`agent_configs.show_avatar`） | 头像显隐是界面偏好，对全部人格统一生效，不必每卡配 |
| 人格 vs 技能 | 人格管「我是谁」，技能管「干什么」，叠加 | 职责分离；人设稳定、任务可临时挂载切换 |
| 温度归属 | 放人格卡（`temperature`） | 不同人设需要不同发散度（严谨助手低温、创意角色高温），随人格走最自然 |
| 首张卡 | 自动设为生效 | 避免「有卡但无生效人格」的空窗 |
| 删/改 | 物理删除；部分更新 `exclude_unset` | 个人项目数据量小，物理删够用；部分更新避免漏字段被清空 |

## 五、易踩坑点

1. **互斥更新必须同事务**：`deactivate_all` 故意不提交，要和目标卡的 `is_active=True` 一起 `commit`。如果各自提交，中途崩溃会出现「零条生效」或并发下「两条生效」。

2. **删除当前生效卡会留下「无生效人格」**：当前实现删卡不自动改选其他卡为生效。此时 `get_active` 返回 `None`，对话兜底温度 `0.7`、空人设。这是可接受的降级（不报错），但前端要意识到「可能没有生效人格」。

3. **`avatar_key` 的「移除」语义**：`PersonaUpdate` 里 `avatar_key` 传空串表示移除头像（service 里 `body.avatar_key or None` 落 `None`）。注意与「未传该字段」区分——`update` 对 `avatar_key` 用 `"avatar_key" in fields` 判断是否要改，而不是判断真值。

4. **头像 URL 失败不能炸列表**：`to_out_dict` 里换 URL 包了 try/except，单个头像取 URL 失败只 warning，仍返回其余字段。否则一个坏 key 会让整个人格列表接口 500。

5. **温度在「建模型时」就要传**：`temperature` 影响的是模型实例的采样参数，必须在 `build_default_chat_model(..., temperature=...)` 时传入，而不是注入到 prompt 文本里。

6. **`show_avatar` 默认关**：`agent_configs.show_avatar` 默认 `False`。新用户默认不显示头像，要显示需显式开启。这是产品默认，别误以为「配了头像就一定显示」。

## 六、面试问答（八股）

**Q1：为什么人格用独立表，而不是在 `agent_configs` 里加几个字段？**

`agent_configs` 是「每用户一条」的全局配置（工具开关、主动召回、跨会话、头像开关），它表达的是「这个用户的助手怎么运转」。而人格是「一对多」的——用户要维护多张可切换的角色卡，这是集合语义，天然该用独立表 `agent_personas`（每用户多行 + 一条 `is_active`）。把多行塞进单行配置表既不优雅也无法表达「多组并存、互斥生效」。两张表分工清晰：人格管「我是谁」，`agent_configs` 管「行为开关」，对话时叠加。

**Q2：怎么保证「同一时刻只有一个人格生效」？为什么不用数据库唯一索引？**

靠 service 层的原子事务：`activate` 里先 `deactivate_all`（`UPDATE ... SET is_active=false WHERE user_id=? AND is_active` 不提交），再把目标卡 `is_active=True`，最后**一起 commit**，中途异常整体回滚。没用「部分唯一索引（`WHERE is_active=true`）」是因为切换过程中会短暂存在「即将更新的旧生效卡 + 新生效卡」的中间态，部分唯一索引会在那一刻报冲突、处理起来反而麻烦；而应用层串行事务简单、可控、够用（个人项目无高并发切换）。

**Q3：头像为什么存 key 不存 URL？**

对象存储（OSS / 本地存储）的访问 URL 往往是**带签名、会过期**的临时地址，或者随存储后端迁移而改变。如果库里存 URL，过期/迁移后历史数据全失效。所以库里只存稳定不变的 `avatar_key`（存储内的相对路径/键），在出参时用当前存储后端 `get_url(key)` 实时换成可访问 URL。这样无论 URL 规则怎么变、存储怎么迁移，库里的 key 都不用动。

**Q4：人格的 `temperature` 是怎么生效的？注入 prompt 吗？**

不是注入 prompt，而是作为**模型采样参数**在构建模型实例时传入：`build_default_chat_model(session, user_id, temperature=persona.temperature, ...)`。温度控制的是 LLM 解码时的随机性（越高越发散），属于推理参数不是文本内容。把温度随人格走，是因为不同人设需要不同发散度——严谨助理用低温更稳定，创意/角色扮演用高温更灵动。没有生效人格时兜底 `0.7`。

**Q5：人格人设和技能提示词是怎么叠加的？冲突了听谁的？**

在 `_compose_system_prompt(persona, skill)` 里拼接：人格人设放最前（定「我是谁」），技能任务提示词以 `【当前任务能力：xxx】` 段追加在后（定「现在干什么」），再拼 few-shot 示例。它们是**互补叠加**而非互斥：人格给语气和身份，技能给任务方法和输出格式。实践中很少冲突；真冲突时，靠 prompt 的措辞引导（技能提示词聚焦任务、明确不写人设，见技能优化的元提示词约定），让模型「以这个身份去做这个任务」。

**Q6：删掉当前生效的人格会怎样？**

当前实现是物理删除，且不自动改选其他卡接替生效。删掉生效卡后，`get_active` 返回 `None`，对话时走兜底：温度 `0.7`、空人设（system prompt 里人格部分为空，但技能/日期块/召回仍照常）。不会报错、对话照常进行，只是「没有特定人设」。这是有意的降级设计——不强行替用户选一个人格。前端可在没有生效人格时提示用户去选一个。

**Q7：头像显隐为什么是用户级开关而不是每张卡一个开关？**

头像显隐是**界面偏好**——用户要么喜欢看头像气泡（像聊天软件），要么喜欢纯文本对话，这个偏好对所有人格是统一的，没有「这张卡显示、那张卡不显示」的实际需求。所以放在每用户一条的 `agent_configs.show_avatar`，一处控制全局。开启时显示 AI 人格头像 + 用户头像，关闭时两边都不显示。默认关。

**Q8：人格列表接口里某个头像的 URL 生成失败，会影响整个列表吗？**

不会。`to_out_dict` 里换 URL 的逻辑包在 try/except 中，单个头像 `get_url` 失败只记 `logger.warning` 并把 `avatar_url` 留空，其余字段照常返回，循环继续处理其他人格。这遵循项目「副作用失败不阻断主流程、能降级就降级」的健壮性原则——一个坏掉的头像 key 不该让整个人格列表接口 500。
