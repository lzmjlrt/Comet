# AI 主动关心 — 设计与八股（后端）

> V0.0.3 情感化体验特性。在「今日回顾」的基础上增强：除了客观汇总当天动态，再加一句**前瞻关怀句**——像一个惦记着你的朋友，结合你的情绪、洞察、最近在意的事，主动说一句温暖的关心或提醒；并支持点「聊聊」直接带着这个话题开聊（AI 主动开场，模型能接住）。本篇只讲后端。

---

## 一、功能定位与需求

「今日回顾」原本只是客观简报（今天提了几次问、记了几件事、读了几份文档）。AI 主动关心给它注入「情感温度」：

- **加一句前瞻关怀**：`daily_reviews` 表加 `care` 字段，存一句基于情绪 + 洞察 + 最近在意的事归纳出的主动关心/提醒（≤40 字），如「Rust 学得怎么样啦，别给自己太大压力～」。
- **消费多源信号**：关怀句结合当天情绪画像（valence-arousal）+「AI 眼中的你」洞察 top-3 + 当天记忆/提问，让关心有针对性而非套话。
- **两次 LLM 调用并行**：回顾正文和关怀句是两次独立 LLM 调用，串行会让仪表盘加载明显变慢，用 `asyncio.gather` 并行优化。
- **点「聊聊」主动开场**：用户点关怀条的「聊聊」会带一句 AI 开场白进入新会话——开场白先作为 assistant 消息落库进历史，这样模型回复时能「接住」这个话题，而不是凭空续写。
- **方案乙**：关怀句融进今日回顾卡片（关怀条），不单独做一张卡片。

## 二、数据模型与迁移

### 2.1 `care` 字段（`api/app/models/daily_review_model.py`）

`DailyReview` 表（每用户每天一条，`UniqueConstraint(user_id, review_date)`）加一列：

| 字段 | 类型 | 说明 |
|------|------|------|
| `care` | `Text`，可空 | 前瞻关怀句：基于情绪+记忆+洞察生成的一句主动关心/提醒，可点「聊聊」开聊 |

可空是因为：数据为空（当天无动态）或 LLM 生成失败时，关怀句允许缺省，不阻断回顾正文。

### 2.2 迁移（`api/app/migrations/versions/e33f64acf851_daily_review_add_care.py`）

`care` 是可空 Text 列，直接加列即可，无需 `server_default` 回填（NULL 是合法值）：

```python
def upgrade():
    op.add_column('daily_reviews', sa.Column('care', sa.Text(), nullable=True))
```

这与前面几个非空布尔开关列形成对比——可空列加列最简单，不存在「存量行无值」的问题。

## 三、核心实现与代码路径

分层：今日回顾接口（`dashboard_controller` 的 `/dashboard/daily-review`，按需生成）+ beat 定时（`tasks/beat.py`）→ `services/daily_review_service.py`（采集 + 双 LLM 并行生成）→ prompt `core/dashboard/prompts/daily_care.jinja2`；「聊聊」开场白落库在 `chat_controller.py` + `chat_service.py`。

### 3.1 采集多源信号（`api/app/services/daily_review_service.py`）

`_collect` 取当天的用户提问、主动记住、新文档、听过的歌；另有两个专门给关怀句用的采集：

- **`_collect_mood`**：调 `EmotionService.trend(user_id, 1)` 取今天的情绪画像，按效价 valence 转成一句话描述（`>= 0.3` 偏积极、`<= -0.3` 略低落、否则平稳），无情绪记录返回空串。
- **`_collect_insights`**：调 `MemoryGraphRepository().list_insights` 取反思引擎归纳的「AI 眼中的你」洞察，取 top-3 拼成一句话，无则空串。

### 3.2 两次 LLM 调用并行（解决慢）

> 文件：`_generate_content_and_care`

关键性能设计。回顾正文和关怀句是两次独立 LLM 调用，如果串行，仪表盘加载要等两次调用之和。但 `AsyncSession` 非并发安全，不能在并发任务里共用 session。解决办法是**分两段**：先用 session 串行取好所有依赖（client、情绪、洞察），再把两次「纯 httpx、不依赖 session」的 LLM 调用并行：

```python
# 串行取 session 相关数据（不可并发共用 session）
client = await get_optional_client_for_type(self.session, user_id, "chat")
if not client:
    return content_fallback, ""
mood = await self._collect_mood(user_id)
insights = await self._collect_insights(user_id)

# 两次 LLM 调用并行（纯 httpx，无 session 依赖）
content, care = await asyncio.gather(
    self._call_content(client, data, mood, content_fallback),
    self._call_care(client, data, mood, insights),
)
```

这样把「两次 LLM 调用」从串行压缩成并行，仪表盘加载明显变快，又规避了 session 并发踩坑。

### 3.3 关怀句生成（`_call_care` + `daily_care.jinja2`）

`_call_care` 渲染 `daily_care.jinja2`，喂入情绪 + 洞察 top-3 + 最近记忆 top-10 + 最近提问 top-5，`temperature=0.8`（要自然多样）、`max_tokens=200`，生成一句关怀。返回时 `strip('"「」')` 去掉模型可能加的引号。失败返回空串（不阻断回顾正文）。

`daily_care.jinja2` 的要点（对标完整 prompt 规范）：

- **角色与目标**：「你是用户的贴心 AI 伙伴，很懂这个人……写一句主动的关心或提醒，像一个惦记着对方的朋友」。
- **针对性约束**：优先结合「对你的理解」与「未竟的事/最近聊的」给前瞻关怀或轻提醒；结合今天心情（低落给安慰、积极一起高兴）。
- **格式约束**：只输出这一句、不要前缀/引号/Markdown、≤40 字。

### 3.4 正文生成与兜底（`_call_content`）

`_call_content` 渲染 `daily_review.jinja2` 生成正文，`max_tokens=1200`，**带一次重试**（首次失败 `sleep(0.6)` 后重试），两次都失败/为空则用统计兜底文案（「今天有 N 次提问、记住了 M 件事…」）。当天完全无动态（total==0）时直接返回兜底文案 + 空关怀，不调 LLM。

### 3.5 按需生成 + 复用（`get_or_generate`）

每次进仪表盘按当前最新活动重新采集，但避免重复调 LLM：

- 已有非空回顾 **且** 统计数（`_same_stats` 比 messages/memories/documents/songs 四项）没变 → 直接复用，不调 LLM。
- 数据变了或还没有 → 重新生成并覆盖（`content` + `care` + `stats` 一起更新）。

beat 每天 22:00 的 `generate_daily_reviews` 批量为所有用户调 `get_or_generate` 预生成。

### 3.6 「聊聊」→ AI 主动开场（`api/app/services/chat_service.py`）

这是「点击开聊」的后端关键。前端点「聊聊」时带一个 `greeting`（AI 开场白）发起新会话。后端在 `stream_chat` 里：**新会话首轮**（`msg_repo.count(conv.id) == 0`）时，先把 greeting 作为一条 **assistant 消息落库**，再落用户消息：

```python
greeting = (body.greeting or "").strip()
if greeting and await self.msg_repo.count(conv.id) == 0:
    await self.msg_repo.add(
        Message(conversation_id=conv.id, role=ROLE_ASSISTANT, content=greeting)
    )
await self.msg_repo.add(Message(conversation_id=conv.id, role=ROLE_USER, content=user_text, ...))
```

为什么要落库进历史？因为如果开场白只在前端显示、不进对话历史，模型本轮回复时上下文里没有这句话，就会「凭空续写」、接不住话题。把它作为 assistant 消息落库后，它就进了 `_history_messages`，模型能看到「我刚才主动开的场」+「用户的回应」，自然地接着聊。`greeting` 在 `ChatStreamRequest` 里定义（`max_length=2000`），仅 `conversation_id` 为空（新会话）首轮生效。

## 四、设计取舍（已定决策）

| 决策 | 选择 | 理由 |
|------|------|------|
| 呈现方式 | 方案乙：关怀句融进今日回顾卡片 | 不单独做卡片，体验上「回顾 + 关心」一体更自然 |
| 数据落地 | `daily_reviews` 加可空 `care` 字段 | 复用每日回顾表，关怀句缺省可空、不阻断正文 |
| 信号来源 | 情绪 + 洞察 top-3 + 当天记忆/提问 | 多源信号让关怀有针对性，不是套话 |
| 两次 LLM | 取数据串行 + LLM 调用 `gather` 并行 | session 非并发安全只能串行取数据；LLM 纯 httpx 可并行，解决加载慢 |
| 关怀句长度/温度 | ≤40 字，temperature 0.8 | 短而暖、自然多样，像朋友的一句话 |
| 正文生成 | 带一次重试 + 统计兜底 | 正文是回顾主体，重试提升成功率，失败也有兜底文案 |
| 按需生成 | 统计数没变则复用，不重复调 LLM | 进仪表盘高频，避免每次都烧 LLM |
| 开场白 | greeting 作为 assistant 消息**落库进历史** | 模型能接住话题，而非凭空续写 |
| 无动态时 | 直接兜底文案 + 空关怀，不调 LLM | 当天没动态没必要花 LLM 调用 |

## 五、易踩坑点

1. **session 非并发安全是并行优化的前提约束**：不能把「取 client/情绪/洞察」和「LLM 调用」一起塞进 `gather`，因为前者用 `self.session`（PG 查询），并发会炸。必须先串行取完所有 session 依赖，再并行那两次纯 httpx 的 LLM 调用。顺序不能乱。

2. **greeting 必须在「新会话首轮」才落库**：判断条件是 `greeting and msg_repo.count(conv.id) == 0`。如果不判空会话，老会话再带 greeting 会重复插入开场白污染历史。只有全新空会话的第一轮才落这条 assistant 消息。

3. **关怀句要 strip 引号**：LLM 常给关怀句加引号/书名号，`_call_care` 返回时 `strip('"「」')` 去掉，否则前端展示会带多余引号。prompt 里也明确要求不要引号，双保险。

4. **care 可空、生成失败不阻断正文**：关怀句是增强，`_call_care` 失败返回空串，回顾正文照常生成落库。别把关怀句失败处理成整个回顾失败。

5. **按需生成的「复用」靠统计数比对**：`_same_stats` 比的是 messages/memories/documents/songs 四项计数。只要计数没变就复用旧回顾不调 LLM——这意味着「内容变了但条数没变」的极端情况不会重新生成，是有意的性能取舍（今日回顾不追求实时精确）。

6. **情绪/洞察采集失败要降级空串**：`_collect_mood` / `_collect_insights` 各自 try/except 返回空串，prompt 里用「（暂无情绪数据）」「（暂无）」兜底占位，保证关怀句仍能生成（只是少了针对性），不会因情绪/洞察拿不到就整体失败。

7. **正文 max_tokens 1200、关怀 max_tokens 200**：两次调用参数不同——正文要展开（1200），关怀要短（200 + prompt ≤40 字约束）。别共用一套参数。

## 六、面试问答（八股）

**Q1：AI 主动关心是什么？和原来的今日回顾什么关系？**

今日回顾原本是客观简报——汇总当天提了几次问、记了几件事、读了几份文档。AI 主动关心是在它基础上加「情感温度」：`daily_reviews` 加一个 `care` 字段，存一句基于用户情绪、AI 对用户的洞察、最近在意的事生成的主动关心或提醒（≤40 字），像惦记着你的朋友说的一句话。呈现上用方案乙——关怀句融进今日回顾卡片，不单独做卡片。还支持点「聊聊」带着这个话题直接开聊。

**Q2：关怀句怎么做到「有针对性」而不是套话？**

靠多源信号 + 专门的 prompt。信号来源：① 当天情绪画像（valence-arousal，转成「偏积极/略低落/平稳」一句话）；② 反思引擎归纳的「AI 眼中的你」洞察 top-3；③ 当天的记忆和提问。`daily_care.jinja2` 让模型「优先结合对用户的理解和未竟的事给前瞻关怀，结合今天心情——低落给安慰、积极一起高兴」。这样关怀句能说出「Rust 学得怎么样了」这种针对性的话，而不是泛泛的「今天辛苦了」。

**Q3：为什么会有两次 LLM 调用？怎么解决加载慢的问题？**

回顾正文是一次调用（展开的简报），关怀句是另一次调用（一句话关心），两者 prompt、温度、max_tokens 都不同，所以分两次。串行的话仪表盘要等两次之和，明显慢。优化是分两段：先用 AsyncSession 串行取好所有依赖（chat client、情绪、洞察）——因为 session 非并发安全不能并发用；再把两次「纯 httpx、不依赖 session」的 LLM 调用用 `asyncio.gather` 并行跑。这样两次调用变并行，加载快了，又规避了 session 并发问题。

**Q4：点「聊聊」之后 AI 怎么「主动开场」？为什么开场白要落库？**

前端点「聊聊」时带一个 greeting（AI 开场白）发起新会话。后端在 stream_chat 里判断是新会话首轮（消息数为 0），先把 greeting 作为一条 **assistant 消息落库**，再落用户消息。落库的原因是：如果开场白只在前端显示不进历史，模型本轮回复时上下文里没这句话，就会凭空续写、接不住。落库后它进了对话历史，模型能看到「我刚主动开的场 + 用户的回应」，自然地接着聊。greeting 只在新会话（conversation_id 为空）首轮生效。

**Q5：care 字段为什么是可空的？迁移和那些布尔开关列有什么不同？**

care 可空是因为它是增强信息——当天无动态、或 LLM 生成关怀句失败时，允许它是 NULL，不阻断回顾正文。迁移上它比布尔开关列简单：可空 Text 列直接 `add_column(nullable=True)` 即可，NULL 是合法默认值，不存在「存量行无值」的问题。而那些非空布尔开关列（enable_active_recall 等）必须先 `server_default` 回填存量行再清除默认，否则加列会失败。

**Q6：每次进仪表盘都重新生成回顾吗？怎么控成本？**

不是每次都重新生成。`get_or_generate` 每次按当前活动重新采集统计，但靠 `_same_stats` 比对四项计数（提问/记忆/文档/听歌）：已有非空回顾且计数没变就直接复用旧的、不调 LLM；计数变了或还没有才重新生成覆盖。仪表盘是高频入口，这个复用机制避免每次进都烧两次 LLM 调用。另外 beat 每天 22:00 还会批量预生成一次。

**Q7：关怀句生成失败了会怎样？**

关怀句是增强，失败不阻断主体。`_call_care` 整段 try/except，失败返回空串，回顾正文照常生成落库，前端就是不显示关怀条。情绪/洞察采集失败也各自降级空串，prompt 里用「（暂无）」占位，关怀句仍能生成只是少了针对性。回顾正文则更稳——带一次重试，两次都失败用统计兜底文案。整体遵循「主体优先、增强可降级」。

**Q8：关怀句为什么限制 40 字、还要 strip 引号？**

40 字是为了「像朋友的一句话」——长了就变说教、罗列数据，失去关心的味道。prompt 里明确约束 ≤40 字、不要前缀、不要引号、不要 Markdown。但中文模型常不听话给关怀句套上引号或书名号，所以代码侧 `_call_care` 返回时再 `strip('"「」')` 去一遍引号，prompt 约束 + 代码兜底双保险，保证前端展示干净。
