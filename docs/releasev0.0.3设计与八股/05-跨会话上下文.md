# 跨会话上下文 — 设计与八股（后端）

> V0.0.3 体验增强特性。让 AI「记得你在别的会话聊过什么」：开启后，每轮对话把最近几个**其他会话**的标题 + 最后几轮原始对话片段拼成一段背景，追加到 system prompt，使跨会话也能自然接着聊。开关默认关。本篇只讲后端。

---

## 一、功能定位与需求

对话默认是「会话隔离」的：每个会话的上下文只有自己的历史消息，换个会话就「失忆」。但用户常有这样的诉求——昨天在 A 会话聊了一个项目，今天在 B 会话想接着问，希望 AI 还记得。跨会话上下文补的就是这一层：

- **注入最近其他会话片段**：取最近 3 个**其他**会话（排除当前会话），每个会话取最后 4 轮对话，拼成「最近你还和我聊过」背景块注入 system prompt。
- **注入原始片段，不萃取**：直接给「用户说了什么、我答了什么」的原片段，而不是抽象成事实。这是它与记忆主动召回最本质的区别。
- **纯 PostgreSQL 实现**：数据就是 conversations / messages 两张已有表，无需建表、无需向量索引（做法 A）。
- **开关默认关**：`agent_configs.enable_cross_session` 默认 `false`。跨会话注入是「双刃剑」（可能把不相关会话的内容带进来），所以默认关，用户按需开。
- **失败/无其他会话返回空串**：纯旁路增强，绝不影响对话主流程。

---

## 二、数据模型与迁移

跨会话上下文**不建任何新表、不建向量索引**（做法 A），数据完全来自既有的 `conversations`（会话，带 `title` / `updated_at`）和 `messages`（消息，带 `role` / `content` / `created_at`）。只在 `agent_configs` 加一个开关字段。

### 2.1 开关字段（`api/app/models/agent_config_model.py`）

`AgentConfig` 加一列：

| 字段 | 类型 | 默认 | 说明 |
|------|------|------|------|
| `enable_cross_session` | `Boolean` | `False` | 是否注入最近其他会话的摘要 |

### 2.2 迁移（`api/app/migrations/versions/db88b5cfcf7b_agent_config_add_enable_cross_session.py`）

非空布尔列加列，沿用「先 `server_default` 回填，再清除默认」套路，但默认值是 `false`（默认关）：

```python
op.add_column(
    'agent_configs',
    sa.Column('enable_cross_session', sa.Boolean(), nullable=False, server_default=sa.false()),
)
op.alter_column('agent_configs', 'enable_cross_session', server_default=None)
```

先用 `server_default=sa.false()` 让存量行回填 `false`（即存量用户默认关闭，行为不变），加列后清除默认，新行默认由 ORM 层（`default=False`）控制。

## 三、核心实现与代码路径

实现集中在 `chat_service.py` 一个方法 `_cross_session_context`，配合既有的 `conversation_repository` 查询，没有专门的 core 模块——因为做法 A 足够轻量，纯 PG 查询 + 字符串拼装。

### 3.1 调用点（`api/app/services/chat_service.py`）

`stream_chat` 组装 system prompt 时，开关开才注入（注意 `agent is not None and ...`，与主动召回的 `agent is None or ...` 相反，体现默认关）：

```python
# 跨会话上下文：注入最近其他会话摘要（默认关，开关控制）
if agent is not None and agent.enable_cross_session:
    cross = await self._cross_session_context(user_id, conv.id)
    if cross:
        system_prompt = (system_prompt + "\n\n" + cross).strip()
```

### 3.2 `_cross_session_context` 实现

```python
async def _cross_session_context(self, user_id, current_conv_id) -> str:
    try:
        convs = await self.conv_repo.list_by_user(user_id)            # 按 updated_at 倒序
        others = [c for c in convs if c.id != current_conv_id][:3]    # 排除当前会话，取最近 3 个
        if not others:
            return ""
        blocks = []
        for c in others:
            msgs = await self.msg_repo.recent_history(c.id, 4)        # 每会话最后 4 轮
            if not msgs:
                continue
            lines = [f"〔会话「{c.title}」〕"]
            for m in msgs:
                if m.role == ROLE_USER:
                    lines.append(f"用户：{(m.content or '').strip()}")
                elif m.role == ROLE_ASSISTANT:
                    lines.append(f"我：{(m.content or '').strip()}")
            blocks.append("\n".join(lines))
        if not blocks:
            return ""
        block = "【最近你还和我聊过（仅供参考，不必主动提起）】\n" + "\n\n".join(blocks)
        if len(block) > cross_session_max_chars:                      # 1200 字兜底截断
            block = block[:cross_session_max_chars] + "…"
        return block
    except Exception:
        return ""   # 失败/无其他会话返回空串，不影响对话
```

要点：

- **排除当前会话**：`c.id != current_conv_id`，否则会把当前会话历史重复注入（当前会话历史已通过 `history` 进了上下文）。
- **取「最近」靠 `updated_at` 排序**：`list_by_user` 按 `updated_at desc` 排序，每轮对话 `touch` 会更新活跃时间，所以「最近聊过的」会话排在前面，取前 3 个。
- **「最后几轮」靠 `recent_history`**：取最近 `max_turns * 2` 条消息（user/assistant 成对）按时间正序返回，所以拿到的是该会话最后 4 轮的原始片段。
- **角色映射成自然口吻**：`ROLE_USER → 「用户：」`、`ROLE_ASSISTANT → 「我：」`，让模型读起来像「我和用户之前的对话记录」。
- **背景块措辞弱化指令感**：「仅供参考，不必主动提起」，避免模型生硬复述「你昨天说过…」。
- **长度兜底**：超 1200 字截断，防止多个会话片段堆太长挤占上下文。

### 3.3 配置项（`api/app/config.py`）

```python
cross_session_max_convs: int = 3        # 取最近几个其他会话
cross_session_turns_per_conv: int = 4   # 每会话取最后几轮
cross_session_max_chars: int = 1200     # 注入上限
```

## 四、设计取舍（已定决策）

| 决策 | 选择 | 理由 |
|------|------|------|
| 实现方式 | 做法 A：纯 PG，不建表/不建向量索引 | 数据就是现成的 conversations/messages，轻量到位，无需额外基础设施 |
| 注入内容 | 最近 3 个其他会话 × 最后 4 轮**原始片段** | 给原对话片段而非萃取事实，保留「接着聊」的具体语境 |
| 取「最近」 | 按会话 `updated_at` 倒序 | 每轮 touch 更新活跃时间，最近聊的排前面，最贴近用户当下心智 |
| 开关 | `enable_cross_session` 默认**关** | 跨会话注入可能带入不相关内容，是双刃剑，默认关、按需开 |
| 排除当前会话 | 过滤 `current_conv_id` | 当前会话历史已在上下文里，避免重复注入 |
| 长度上限 | 1200 字截断 | 多会话片段堆叠易过长，兜底防挤占上下文 |
| 与召回的关系 | 注入**原片段** vs 召回注入**萃取事实** | 定位互补：跨会话给近期对话语境，召回给长期画像事实 |
| 失败处理 | try/except 返回空串 | 旁路增强，失败不阻断对话 |

## 五、易踩坑点

1. **务必排除当前会话**：`list_by_user` 返回的会话列表包含当前会话，必须 `c.id != current_conv_id` 过滤掉，否则当前会话的历史会被重复注入一次（它已经通过正常 history 进上下文了）。

2. **「最近」依赖 `touch` 更新 `updated_at`**：会话排序靠 `updated_at desc`，而 `updated_at` 是每轮对话结束 `conv_repo.touch` 更新的。如果某条路径漏调 touch，会话排序会失真，「最近聊过」就不准。

3. **`recent_history` 的轮数是 `max_turns * 2` 条消息**：参数语义是「轮」，内部取 `max_turns * 2` 条（user + assistant 成对）并 reverse 成正序。别误以为传 4 就是 4 条消息——是 4 轮 ≈ 8 条。

4. **开关判断方向和主动召回相反**：跨会话是 `agent is not None and agent.enable_cross_session`（默认关，agent 为 None 时不注入）；主动召回是 `agent is None or agent.enable_active_recall`（默认开）。两个开关默认值不同，判断写法要对应，别抄错。

5. **原片段可能很长**：注入的是原始对话内容，单条消息可能很长（用户贴了大段文字）。靠 1200 字总长度兜底截断防爆，但极端情况下一个会话的一条长消息就可能占满预算，注意这是「尽力而为」的背景，不保证完整。

6. **纯 PG 查询要控量**：取 3 个会话各查一次 `recent_history`，是 1 + 3 = 4 次 PG 查询，量小。但因为和主动召回、工具构建共用同一个 AsyncSession，是串行执行的，别在这里加更多会话数让它变慢。

## 六、面试问答（八股）

**Q1：跨会话上下文是什么？解决什么问题？**

对话默认按会话隔离，每个会话只看得到自己的历史，换个会话 AI 就「失忆」。但用户常想「昨天在另一个会话聊的事，今天接着问」。跨会话上下文就是：开启后每轮对话把最近 3 个其他会话的标题 + 最后 4 轮原始片段拼成背景块注入 system prompt，让 AI 跨会话也能接着聊。开关默认关，按需开。

**Q2：为什么用「做法 A」纯 PG，不建表也不建向量索引？**

因为需要的数据本来就在 conversations 和 messages 两张现成表里——会话标题、活跃时间、消息内容都有。要做的只是「取最近几个会话的最后几轮」，这是简单的排序 + limit 查询，纯 PG 几次查询就够。建专门的表或向量索引属于过度设计：跨会话注入的是「最近聊的原片段」，按时间取就行，不需要语义检索。轻量到位是这个特性的设计基调。

**Q3：跨会话上下文和记忆主动召回有什么区别？两个不是重复了吗？**

不重复，是两种互补的「让 AI 记得你」。区别在注入的**内容形态**和**时间维度**：① 跨会话注入的是**最近其他会话的原始对话片段**（用户说了啥、我答了啥），保留具体语境，偏「近期、短时」；② 主动召回注入的是从全部历史**萃取并归纳的事实和洞察**（实体、关系、画像结论），是抽象后的长期记忆，偏「长期、稳定」。一个给「近期对话的来龙去脉」，一个给「关于你这个人的沉淀认知」。所以可以同时开，分别补不同维度。

**Q4：怎么取到「最近的其他会话」和「最后几轮」？**

「最近的其他会话」：`list_by_user` 按 `updated_at desc` 排序返回，过滤掉当前会话 id，取前 3 个。`updated_at` 由每轮对话结束时的 `touch` 更新，所以排在前面的就是最近活跃的会话。「最后几轮」：对每个会话调 `recent_history(conv_id, 4)`，它取最近 `4*2` 条消息（user/assistant 成对）按时间倒序，再 reverse 成正序，得到该会话最后 4 轮的原始片段。

**Q5：为什么默认关？**

跨会话注入是双刃剑。好处是跨会话能接着聊；风险是会把「其他不相关会话」的内容带进当前对话，可能干扰回答、甚至泄露用户在另一话题下的内容到当前语境。比如用户在 A 会话聊私密话题、B 会话聊工作，自动把 A 的片段注入 B 未必合适。所以默认关，让用户明确知道并主动开启。主动召回是经过萃取的稳定事实、噪声可控，所以默认开——两个开关默认值不同正是基于风险差异。

**Q6：注入失败或没有其他会话会怎样？**

返回空串，对话照常。`_cross_session_context` 整段 try/except，没有其他会话（新用户只有当前会话）返回空串，查询异常也返回空串并 warning。调用点判断 `if cross:` 才追加，空串不注入。这是旁路增强，失败绝不阻断主对话，遵循「副作用失败不阻断主流程」原则。

**Q7：注入的内容怎么避免让模型生硬地复述「你昨天说过…」？**

靠背景块的措辞设计。块标题写「【最近你还和我聊过（仅供参考，不必主动提起）】」，明确告诉模型这是参考语境、不要刻意提起。角色也映射成自然口吻（用户：/我：），让模型读起来像「我和用户之前的对话记录」而不是一份指令。措辞直接影响体验，是 prompt 工程的一部分。

**Q8：性能上有什么考量？**

做法 A 本身很轻：1 次查会话列表 + 最多 3 次查各会话最近消息 = 4 次 PG 查询，都是带索引的简单查询。注意它和主动召回、工具构建共用同一个 AsyncSession（非并发安全），是串行执行的，所以不在这里堆更多会话数。另外做了 1200 字总长度兜底截断，防止多会话原片段堆叠过长既挤占上下文又增加 token 成本。
