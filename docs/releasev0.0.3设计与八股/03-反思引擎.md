# 反思引擎 Reflection — 设计与八股（后端）

> V0.0.3 记忆增强特性。在原子记忆（实体 + 陈述）之上再叠一层「画像级理解」：定期回看一个人最重要、最高频的实体与代表性陈述，用 LLM 归纳出三到六条「这个人是谁、在意什么、处于什么状态」的高层洞察 Insight，落成 Neo4j 图节点并向量化，供记忆主动召回按话题检索。本篇只讲后端。

---

## 一、功能定位与需求

记忆系统底层是「四层溯源图谱」：`Dialogue → Chunk → Statement → Entity`，萃取出的是一条条原子事实（「在腾讯做后端」「在学 Rust」「养了只叫多多的小狗」）。这些事实足够细，但缺一个「站在更高处看人」的视角——AI 没法直接说出「这是一位有上进心、持续精进的后端工程师」。反思引擎补的就是这一层：

- **归纳高层洞察**：回看用户记忆里 top-25 高重要度/高频的实体 + 每个实体的代表性陈述，让 LLM 抽象出 3~6 条**画像级结论**（每条归入一个主题 theme，如「职业发展」「学习成长」「生活方式」）。
- **纯 Neo4j 落地**：新增 `Insight` 节点（不动 PostgreSQL，无 PG 迁移），并建 `Insight -[:DERIVED_FROM]-> Entity` 溯源边记录「这条洞察是从哪些实体归纳来的」。
- **向量化供召回**：洞察正文向量化存 `embedding`，建专用向量索引，供「记忆主动召回」每轮按当前问题检索相关洞察注入对话。
- **双触发**：Celery beat 每天 4:30 全量反思 + 萃取攒够 20 个新增实体时 Redis 计数增量触发单用户反思。
- **遵循「记忆不遗忘」**：反思只新增/更新高层洞察节点，绝不删改底层原子记忆。

定位上，反思引擎是「消费者」也是「生产者」：它消费记忆萃取沉淀的实体与陈述，生产出洞察给记忆主动召回（③）和 AI 主动关心（⑧）消费。所以它在开发顺序上排在召回之前。

---

## 二、数据模型与迁移

**关键点：反思引擎不碰关系型数据库，所有数据都在 Neo4j。** 因此没有 Alembic 迁移，只有图 schema（约束/索引/向量索引）的幂等创建。

### 2.1 Insight 节点（`api/app/core/memory/graph_models.py`）

`InsightNode` 字段：

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | str | 节点唯一 id（hex） |
| `user_id` | str | 多租户隔离 |
| `theme` | str | 主题标签（如「职业发展」「兴趣偏好」），**收敛键** |
| `content` | str | 洞察正文（一句话高层归纳结论） |
| `embedding` | list[float] \| None | 正文向量，供按话题召回 |
| `importance` | float | 重要度（LLM 评分，默认 0.6） |
| `confidence` | float | 置信度（LLM 评分，默认 0.7） |
| `source_count` | int | 归纳自多少个实体（= DERIVED_FROM 边数） |
| `created_at` / `updated_at` | datetime | 创建/更新时间 |

新增关系类型常量 `REL_DERIVED_FROM = "DERIVED_FROM"`，对应 `DerivedFromEdge`（`Insight → Entity`），记录洞察的归纳依据，可溯源。

### 2.2 图 schema（`api/app/core/memory/graph_schema.py`）

`ensure_graph_schema()` 在应用启动 lifespan 幂等执行，为 Insight 加了三类索引：

```python
# 唯一约束（保证 MERGE 幂等）
"CREATE CONSTRAINT insight_id_unique IF NOT EXISTS FOR (n:Insight) REQUIRE n.id IS UNIQUE"

# 属性索引：按 user_id / theme 检索与收敛
"CREATE INDEX insight_user_idx ... ON (n.user_id)"
"CREATE INDEX insight_theme_idx ... ON (n.theme)"

# 全文索引（cjk 中文分词）
"CREATE FULLTEXT INDEX insight_fulltext ... FOR (n:Insight) ON EACH [n.content, n.theme]
 OPTIONS { indexConfig: { `fulltext.analyzer`: 'cjk' } }"

# 向量索引（维度与 embedding 一致，余弦相似度）
"CREATE VECTOR INDEX insight_embedding_index ... FOR (n:Insight) ON n.embedding
 OPTIONS { indexConfig: { `vector.dimensions`: 1024, `vector.similarity_function`: 'cosine' } }"
```

每条 schema 语句单独 try/except，单条失败（如旧版 Neo4j 不支持向量索引）只 warning 不阻断其余——这样新增 Insight 索引不会因任何一条出错而拖垮整个图 schema 初始化。

## 三、核心实现与代码路径

分层：增量触发在萃取编排器里埋点 → Celery 任务 `tasks/beat.py` → 反思引擎 `core/memory/reflection/reflector.py` → 图仓储 `repositories/neo4j/memory_graph_repository.py` + Cypher `cypher_queries.py` → prompt `prompts/reflect.jinja2`；对外接口在 `controllers/memory_controller.py`。

### 3.1 反思主流程（`api/app/core/memory/reflection/reflector.py`）

`ReflectionEngine.run(user_id)` 的步骤：

1. **前置门控**：`chat_client` 为空（未配 chat 模型）直接跳过返回 `{"skipped": "no_chat_model"}`；取出 top-25 实体后若数量 `< reflection_min_entities`（默认 5）也跳过（信息太少不值得归纳）。
2. **取反思输入**：`repo.reflection_top_entities(user_id, top_k=25)` 按 `importance DESC, mention_count DESC` 取高价值实体；对每个实体再取 `reflection_entity_statements(...)` 按重要度倒序的代表性陈述（每实体 4 条）。
3. **组装记忆清单**：`_build_memory_block` 把实体（类型/描述/核心事实/特质）+ 代表性陈述拼成可读的中文清单文本，同时返回 `name_to_id` 映射（洞察溯源用）。
4. **LLM 归纳**：渲染 `reflect.jinja2`，`temperature=0.5, max_tokens=2048` 调用 chat 模型；返回值用 `parse_json_object`（json_repair 健壮解析）解析出 `insights` 列表。
5. **向量化**：把每条洞察的 `content` 批量 `embed_texts`（embed_client 为空则洞察不带向量，仍可落库）。
6. **按 theme upsert 落库**：逐条校验 theme/content 非空，调 `repo.upsert_insight(...)`，单条失败只 warning 跳过。

### 3.2 按 theme 收敛（upsert，而非每次堆叠）

> 文件：`memory_graph_repository.upsert_insight` + Cypher `INSIGHT_GET_BY_THEME` / `INSIGHT_UPSERT`

反思是周期性反复跑的，如果每次都新建节点，「职业发展」会越堆越多。所以洞察以 **theme 为收敛键**：upsert 前先 `INSIGHT_GET_BY_THEME` 查同 user + 同 theme 的已有节点，命中就复用其 id 更新内容，没有才新建。配合 prompt 里「同一主题只输出一条最具代表性的洞察」，保证每个主题始终只有一条最新的画像结论。

```python
rec = await r.single()            # 查同主题已有洞察
insight_id = rec["id"] if rec else _uuid.uuid4().hex
await session.run(cq.INSIGHT_UPSERT, id=insight_id, ...)   # 按 id MERGE
```

### 3.3 溯源边「先清后建」

每次 upsert 后，DERIVED_FROM 边都重建：先 `INSIGHT_CLEAR_DERIVED` 删掉该洞察的所有旧来源边，再 `INSIGHT_LINK_ENTITIES` 按本轮 `based_on`（LLM 给的依据实体名，经 `name_to_id` 映射成 id）建新边。这样洞察的来源始终反映「最近一次归纳依据的实体」，不残留过时边。`source_count` 取本轮有效实体 id 数。

### 3.4 content 截断三重兜底

洞察是「一句话概括」，必须短，否则注入对话会喧宾夺主、也容易出现截断残句。做了三层兜底：

1. **prompt 约束**：`reflect.jinja2` 明确「content 必须是一句简洁的高层概括，控制在 50 字以内，不要堆砌细节流水」。
2. **max_tokens 限制**：LLM 调用 `max_tokens=2048`，限制整体输出体量。
3. **落库前硬截断**：`if len(content) > 200: content = content[:200].rstrip() + "…"`——即使模型不听话写了长文，也兜底截到 200 字，避免异常长文/截断残句入库。

`importance` / `confidence` 经 `_clamp` 夹到 `[0,1]`，非法值用默认（0.6 / 0.7）。

### 3.5 双触发机制

**触发一：定时全量（`api/app/celery_app.py` + `api/app/tasks/beat.py`）**

beat 配置每天 4:30 跑 `reflect_memory`（特意排在 4:00 记忆巩固之后——先巩固提升重要度，再基于更准的重要度反思）：

```python
"reflect-memory": {
    "task": "app.tasks.beat.reflect_memory",
    "schedule": crontab(hour=4, minute=30),  # 每天凌晨 4:30 反思（巩固之后）
}
```

`reflect_memory_task` 遍历所有用户，逐个取其 chat / embedding 客户端构 `ReflectionEngine` 跑 `run`，单用户失败只 warning。

**触发二：萃取增量触发（`api/app/core/memory/extraction/orchestrator.py`）**

每次记忆萃取写图完成后，用 Redis 计数器累计新增实体数，达阈值（`reflection_trigger_threshold=20`）就清零并派发单用户反思任务：

```python
key = f"reflect:pending:{user_id}"
total = await redis.incrby(key, new_count)
if total >= settings.reflection_trigger_threshold:
    await redis.set(key, 0)
    reflect_user_task.delay(user_id)   # 派发单用户反思
```

整个增量触发包在 try/except 里，失败只 warning，绝不影响萃取主流程。这样既有「每日兜底」，又能在用户活跃产生大量新记忆时「及时跟进反思」。

### 3.6 对外接口（`api/app/controllers/memory_controller.py`）

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/memories/insights` | 列出用户全部洞察（`INSIGHT_LIST` 按重要度+更新时间倒序） |
| POST | `/memories/reflect` | 手动触发一次反思（同步跑 `MemoryService.reflect`） |
| DELETE | `/memories/insights/{insight_id}` | 删除单条洞察（`DETACH DELETE` 连带边） |

另有 `search_insights_by_vector`（`INSIGHT_VECTOR_SEARCH` 走 `insight_embedding_index`）不直接暴露为接口，而是给记忆主动召回内部调用。

### 3.7 配置项（`api/app/config.py`）

```python
reflection_top_k: int = 25            # 反思输入：top-N 高重要度/高频实体
reflection_stmt_per_entity: int = 4   # 每个实体取几条代表性陈述
reflection_min_insights: int = 3      # 期望产出洞察下限
reflection_max_insights: int = 6      # 期望产出洞察上限
reflection_min_entities: int = 5      # 实体少于此数不反思
reflection_trigger_threshold: int = 20  # 增量触发：累计新增实体达标触发
```

## 四、设计取舍（已定决策）

| 决策 | 选择 | 理由 |
|------|------|------|
| 存储位置 | 纯 Neo4j 新增 Insight 节点 | 洞察本质是图谱画像层，复用图的向量/全文索引；无需 PG 表，零迁移 |
| 收敛方式 | 按 theme upsert | 反复跑反思不堆叠，每个主题只留一条最新结论 |
| 溯源 | DERIVED_FROM 边「先清后建」 | 来源始终反映最近归纳依据，可追溯但不残留过时边 |
| 反思输入 | top-25 高重要度+高频实体 + 每实体 4 条代表陈述 | 抓主要矛盾，避免把全部噪声喂给 LLM，省 token 提质量 |
| content 长度 | prompt 50 字 + max_tokens 2048 + 落库 >200 字截断 | 三重兜底保证洞察简洁，注入对话不喧宾夺主 |
| 触发方式 | beat 每天 4:30 + 萃取攒够 20 实体增量触发 | 兜底 + 及时性兼得；增量触发失败不影响萃取 |
| 与巩固的顺序 | 反思排在记忆巩固（4:00）之后（4:30） | 先巩固提升重要度，反思基于更准的重要度排序选实体 |
| 向量化 | 洞察正文 embed + 专用向量索引 | 供主动召回按当前问题语义检索相关洞察 |
| 记忆不遗忘 | 只增/改洞察，不动原子记忆 | 反思是叠加的高层视图，不破坏底层溯源数据 |

## 五、易踩坑点

1. **无 PG 迁移，别去 alembic 找表**：反思引擎所有数据在 Neo4j，新增的是图节点与索引，不存在关系表迁移。Insight 的「schema 变更」体现在 `graph_schema.py` 的约束/索引列表里，靠启动时 `ensure_graph_schema()` 幂等创建。

2. **向量索引可能创建失败要容错**：`insight_embedding_index` 依赖 Neo4j 5.x 的向量索引能力。schema 创建对每条语句单独 try/except，即便向量索引建不出来，洞察仍能落库（只是不带向量、无法被召回）。别把向量索引当硬依赖。

3. **theme 是收敛键，大小写/措辞漂移会导致重复**：upsert 靠 `theme` 完全匹配查已有节点。如果 LLM 这次写「职业发展」下次写「职业 发展」或「事业发展」，会被当成新主题各建一条。靠 prompt 约束主题用词稳定来缓解，必要时可在落库前做主题归一。

4. **embedding 为空时不能漏掉计数对齐**：落库循环里用 `emb_idx` 与 `embeddings` 列表对齐，但 `emb_idx += 1` 是对**所有有效洞察**自增的，而 `embeddings` 只对 `content` 非空的洞察生成——代码里 contents 的过滤条件与落库循环的有效性判断要保持一致，否则向量会错位。当前实现两处都要求 content 非空，是对齐的。

5. **based_on 名字要能映射回实体 id**：LLM 返回的 `based_on` 是实体**名称**，必须经 `name_to_id`（来自本次组装记忆清单时建立的映射）转成 id 才能建 DERIVED_FROM 边。名单外的名字（模型编造）会被过滤掉，不会建边。

6. **反思是周期性后台任务，不要放进对话同步链路**：手动 `/memories/reflect` 是同步跑（用户主动等待），但日常反思走 beat / Celery 异步。LLM 归纳要遍历 top-25 实体 + 各自陈述 + 一次大 LLM 调用，耗时较长，绝不能塞进对话每轮。

7. **增量触发用的 Redis 计数器要清零**：达阈值后必须 `redis.set(key, 0)` 清零再派发，否则会反复触发。计数 key 带 user_id 隔离（`reflect:pending:{user_id}`）。

## 六、面试问答（八股）

**Q1：反思引擎解决了什么问题？和底层记忆萃取是什么关系？**

记忆萃取产出的是原子事实（「在腾讯做后端」「在学 Rust」），细粒度、可溯源，但缺「站在高处看人」的视角。反思引擎在这之上叠一层画像级理解：回看用户最重要/最高频的实体与代表陈述，让 LLM 归纳出「是一位有上进心、持续精进的后端工程师」这类高层洞察。关系上它既是消费者（吃萃取沉淀的实体与陈述），又是生产者（产出洞察给记忆主动召回和 AI 主动关心用）。所以它在开发顺序上排在主动召回之前。

**Q2：为什么洞察存在 Neo4j 而不是新建一张 PG 表？**

洞察本质是记忆图谱的画像层，和实体/陈述同源，存图里能直接复用图的向量索引和全文索引，还能用 DERIVED_FROM 边把洞察和归纳依据的实体连起来做溯源。如果单独建 PG 表，向量召回得再搭一套、溯源得跨库 join，得不偿失。所以反思引擎纯 Neo4j 落地，零 PG 迁移，新增的只是图节点和索引。

**Q3：反复跑反思，洞察会不会越堆越多？怎么收敛？**

按 theme（主题）收敛。每条洞察归入一个主题（职业发展/兴趣偏好…），upsert 前先按 user + theme 查已有节点，命中就复用 id 更新内容、不新建。配合 prompt 里「同一主题只输出一条最具代表性的洞察」，每个主题始终只有一条最新结论。DERIVED_FROM 来源边也是「先清后建」，保证溯源反映最近一次归纳依据，不残留过时边。

**Q4：洞察内容怎么保证简洁、不出现超长或截断残句？**

三重兜底：① prompt 约定 content 控制在 50 字以内、只给结论不堆细节；② LLM 调用 max_tokens=2048 限制整体输出；③ 落库前硬截断，content 超过 200 字就截到 200 字加省略号。即使模型不听 prompt，落库这道也兜得住。重要度/置信度也用 `_clamp` 夹到 [0,1]，非法值走默认。

**Q5：双触发是怎么设计的？为什么要两个触发源？**

定时 + 增量两条。定时：beat 每天 4:30 全量为所有用户反思（特意排在 4:00 记忆巩固之后，先巩固提升重要度再反思）。增量：每次萃取写图后用 Redis 计数器累加新增实体数，达 20 个就清零并派发单用户反思任务。定时保证「每天至少反思一次」的兜底，增量保证「用户活跃产生大量新记忆时能及时跟进」。增量触发整个包在 try/except 里，失败只 warning，绝不影响萃取主流程。

**Q6：反思的输入是怎么选的？为什么不把全部记忆喂给 LLM？**

取 top-25 高重要度/高频实体（`importance DESC, mention_count DESC`），每个实体再配 4 条按重要度倒序的代表陈述。不全量喂是因为：① 全量 token 爆炸、成本和延迟都不可控；② 噪声多反而稀释 LLM 注意力、降低归纳质量。抓「最重要、最高频」的实体就是抓主要矛盾，画像结论的质量更高。实体不足 5 个时直接跳过——信息太少归纳不出有意义的洞察。

**Q7：洞察怎么被对话用起来？为什么要向量化？**

洞察正文落库时向量化存 `embedding`，建专用向量索引 `insight_embedding_index`。记忆主动召回在每轮对话用当前问题的向量去 `search_insights_by_vector` 检索 top-N 相关洞察，拼进「关于用户的已知信息」背景块追加到 system prompt。向量化的意义就是让召回能按「当前话题」语义匹配相关洞察，而不是把全部洞察无差别塞进去。

**Q8：反思失败会影响什么？健壮性怎么做的？**

反思是旁路后台能力，任何失败都不该波及主链路。具体：未配 chat 模型直接跳过；实体太少跳过；LLM 调用失败 / JSON 解析失败返回空洞察并 warning；单条洞察落库失败只 warning 跳过其余照常；增量触发失败 try/except 吞掉不影响萃取；图 schema 单条索引创建失败不阻断其余。JSON 解析统一走 `parse_json_object`（json_repair 修复中文模型常见非法 JSON），不裸 `json.loads`。整体遵循「能降级就降级、副作用失败不阻断主流程」。
