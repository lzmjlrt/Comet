# 划词追问 — 设计与八股（后端）

> V0.0.3 的一个轻量交互特性：用户用鼠标选中 AI 回答里的某个片段，浮出「追问 / 解释」按钮，点一下就把这个片段作为引用填进输入框，继续追问或让 AI 展开解释。**本特性为纯前端实现，没有任何后端改动。** 本篇如实说明「为何无需后端」，以及它如何复用既有的对话接口。

---

## 一、功能定位与需求

- **场景**：AI 的回答往往很长，用户经常只想针对其中某一句、某个名词或某段结论继续追问。手动复制粘贴再补一句话很繁琐。
- **交互**：在 AI 回答气泡内选中一段文字 → 选区附近浮出一个小工具条（「追问」「解释」）→ 点击后，把选中片段以引用形式预填进对话输入框（如「关于「……」，」或「请解释这句：「……」」），光标聚焦、**不自动发送**，等用户补完再发。
- **本质**：它是一个「把选中文本转成一条带引用的普通提问」的前端便捷入口，最终走的还是普通对话流程。

---

## 二、为什么这个功能没有后端改动

判断一个特性要不要后端，看它有没有产生「新的服务端能力或数据」。划词追问全部都没有：

| 维度 | 是否需要后端 | 说明 |
|------|------------|------|
| 新数据要持久化吗？ | 否 | 选区、引用只是前端临时状态，发送后就是一条普通用户消息，沿用 messages 表既有结构 |
| 新接口吗？ | 否 | 追问发送时调用的就是已有的 `POST /chat/stream`，请求体不需要新字段 |
| 新的模型能力 / 提示词吗？ | 否 | 追问就是普通问答，system prompt、工具、记忆全部沿用，不为它特化 |
| 新的鉴权 / 隔离逻辑吗？ | 否 | 还是当前登录用户、当前会话，权限边界不变 |

所以划词追问完全落在前端：监听选区事件（桌面 `mouseup` / 移动端 `touchend`）、判断选区是否落在 AI 回答气泡内（气泡渲染时打了 `data-ai-message` 标记）、浮出工具条、把片段拼成引用文本填进输入框。这些都是浏览器 DOM 层面的事，服务端无从参与，也不需要参与。

---

## 三、它如何复用既有对话接口

> 关键代码路径（后端侧，均为既有、未改动）：`api/app/controllers/chat_controller.py` 的 `POST /chat/stream`、`api/app/services/chat_service.py` 的 `stream_chat`、`api/app/schemas/chat_schema.py` 的 `ChatStreamRequest`。

用户点「追问 / 解释」并补完后点发送时，前端把「引用片段 + 用户补充」拼成一句普通的 `message`，调用既有的流式接口：

```
POST /chat/stream
{
  "conversation_id": "<当前会话>",
  "message": "关于「……AI 原文片段……」，这个结论的依据是什么？"
}
```

这条请求与用户手动敲一句问题**没有任何区别**：

1. `ChatStreamRequest` 的 `message` 字段就是拼好的引用文本，**不需要新增「引用」字段**——引用信息已经内联进自然语言提问里，模型读得懂。
2. `ChatService.stream_chat` 照常处理：确保会话 → 存 user 消息 → 组装 system prompt + 最近 N 轮历史 + 本轮问题 → 按 Agent 配置决定是否走工具编排 → SSE 逐 token 流式 → 存 assistant 消息 → 回答后照常异步萃取记忆。
3. 因为引用片段是作为问题正文的一部分进入历史的，多轮追问时它自然留在上下文窗口里，后续可以接着围绕这个片段继续问。

一句话：**划词只是帮用户「写好一句带引用的问题」，写完之后就是普通对话，后端无需感知「这是一次划词追问」。**

## 四、设计取舍（已定决策）

| 决策 | 选择 | 理由 |
|------|------|------|
| 实现层 | **纯前端** | 选区交互是 DOM 层的事，无新数据/新能力，后端无需参与 |
| 是否新增接口 | **否，复用 `/chat/stream`** | 追问本质是普通问答，新接口只会重复造轮子 |
| 是否在请求体加「引用」字段 | **否** | 引用内联进 `message` 自然语言里，模型可理解，避免协议膨胀 |
| 发送时机 | 填入输入框、**不自动发送** | 让用户补充意图后再发，避免误触发送半句话 |
| 是否落新表 / 改 messages | **否** | 追问产生的就是普通 user 消息，沿用既有 messages 表结构 |

---

## 五、易踩坑点

1. **不要为了「记录这是划词来的」而给后端加字段**：曾考虑在请求里加 `quote` / `source_message_id` 之类的字段，但这会让对话协议为一个纯交互特性买单。引用文本内联进 `message` 已经足够，模型能正确理解「关于「……」」这种引用句式，后端保持零改动最干净。

2. **片段过长要前端截断**：用户可能选中一大段文字。若不加限制，拼进 `message` 会很长、挤占上下文。这是前端的责任（截断 + 省略号），后端 `message` 字段本身没有特别上限，但前端应控制引用片段长度。

3. **气泡标记是前端契约**：选区是否「在 AI 回答内」靠 `data-ai-message` 这个 DOM 标记判断。如果将来改了消息渲染组件，要记得保留这个标记，否则划词工具条不会浮出——但这同样与后端无关。

4. **多轮追问的上下文靠既有历史窗口**：追问能「接着上一句问」是因为引用文本作为普通消息进了历史，落在 `MAX_HISTORY_TURNS` 窗口内。这是既有机制，划词追问没有为它做任何特殊延长。

## 六、面试问答（八股）

**Q1：划词追问后端做了什么？**

后端什么都没做。这是一个纯前端特性：用户选中 AI 回答片段，前端监听选区事件、判断选区落在 AI 气泡内（靠 `data-ai-message` 标记），浮出「追问/解释」工具条，点击后把片段拼成一句带引用的自然语言问题填进输入框。发送时调用的是**既有的** `POST /chat/stream`，请求体和普通提问完全一样。所以没有新表、没有新接口、没有新提示词，后端零改动。

**Q2：为什么不给它单独做一个接口或加请求字段？**

因为划词追问的本质就是「普通对话」——它只是帮用户把一句带引用的问题写好。引用信息内联进 `message` 正文（如「关于「……」，……」），大模型完全能理解这种引用句式，不需要结构化地告诉后端「这是引用、来自哪条消息」。如果为它单加 `quote` 字段或新接口，等于让对话协议为一个交互便捷功能买单，增加维护成本却没带来任何能力，违背「解决问题而不过度设计」的原则。

**Q3：那追问的上下文是怎么保证连贯的？**

完全靠既有机制。引用片段作为用户消息正文存进 messages 表、进入对话历史；`ChatService.stream_chat` 组装上下文时本来就会带最近 N 轮历史（`MAX_HISTORY_TURNS`）。所以围绕同一个片段多轮追问时，引用文本一直在上下文窗口内，模型能接着聊。划词追问没有为连贯性做任何后端特殊处理。

**Q4：如何确认它走的是普通对话流程而不是特殊分支？**

看 `chat_controller.py` 的 `/chat/stream` 与 `ChatService.stream_chat`：入口只有一个 `ChatStreamRequest`，里面没有任何「划词」相关字段；service 的处理逻辑（建会话 → 存 user 消息 → 组装 system+历史+问题 → 分流工具编排 → 流式 → 存 assistant → 异步萃取记忆）对所有提问一视同仁。追问发来的请求和手敲一句问题在后端看来完全相同，不存在划词专用分支。

**Q5：这种「能在前端解决就不动后端」的判断标准是什么？**

看特性有没有产生新的服务端能力或数据：要不要持久化新数据、要不要新接口、要不要新模型能力/提示词、要不要新的鉴权隔离。划词追问四项全是「否」——它只是把选中文本转成一句普通问题。凡是纯交互/纯展示、最终落到既有接口的特性，就应该留在前端，保持后端 API 表面最小、最稳定。
